# Week 9.5 — Multi-tenant model implementation plan

> **For Claude:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Use `superpowers:subagent-driven-development` if dispatching a fresh subagent per task in the same session. Invoke `ax-conventions` and `security-checklist` whenever a task touches a hook surface, IPC boundary, sandbox capability, or new dependency. Tasks marked **[boundary]** require a boundary-review note in the commit message; tasks marked **[security]** require the structured security note.

**Goal:** Introduce user identity, agent entities, and per-agent ACLs — the primitives Week 10–12 channels will sit on top of. Ship admin HTTP endpoints (no UI yet) plus a bootstrap CLI so multi-tenant flows can be exercised end-to-end before the chat UI in Week 10–12.

**Architecture:** Three new plugins (`@ax/http-server`, `@ax/auth`, `@ax/agents`); one optional plugin (`@ax/teams`); two extensions (`@ax/session-postgres` schema bump, `@ax/tool-dispatcher` per-agent filter). Auth runs as middleware on the http-server's request pipeline. Agents are postgres-backed entities with an inline ACL gate on `agents:resolve`. Per-agent tool/MCP scoping is filtered at the dispatcher, not the providers, so MCP-client and tool plugins stay tenant-agnostic. A new `@ax/preset-multi-tenant` (or augmentation of `@ax/preset-k8s`) wires the full stack; `ax-next admin bootstrap` mints the first admin out-of-band.

**Tech Stack:**
- Node.js + TypeScript (existing toolchain)
- Postgres via `@ax/database-postgres` (existing) for new tables
- OAuth/OIDC via `openid-client` (pinned)
- HTTP server via Node `node:http` (no Express — keep deps minimal, mirror `cli/serve.ts` style)
- Cookie signing via Node's built-in `crypto.timingSafeEqual` + HMAC
- Vitest (existing)

**Branch base:** `main` (Week 6.5e MCP-client + Week 7-9 k8s deployment have both landed; commit `b7814f5`). Create the feature branch with `git switch -c feat/week-9.5-multi-tenant` from a clean `main`.

## Invariants for this slice

The earlier slices' rolled-back attempts taught us specific failure modes; we enumerate them as **Iₙ** here so every task can reference them:

- **I1 — Hook surface is transport-agnostic.** No `oauth_subject_id`, `cookie`, `bearer`, `pod_name`, `bucket`, or `pg_*` field names in any hook payload (see `ax-conventions` boundary review).
- **I2 — No cross-plugin imports.** `@ax/auth` does not import `@ax/agents`; both talk through the bus. Enforced by lint after Task 4.
- **I3 — No half-wired plugins.** Each plugin merges with at least one acceptance test that drives a real chat through the new code path. No "wire up later" stubs.
- **I4 — One source of truth per concept.** Users live in `@ax/auth`. Agents live in `@ax/agents`. Sessions live in `@ax/session-postgres`. ACL state never duplicates across these.
- **I5 — Capabilities explicit and minimized.** Every new manifest entry lists the *exact* hooks the plugin calls / registers. No `*` wildcards.
- **I6 — Authz is at the API layer, not UI.** Every admin endpoint enforces ACL even though there's no UI in this slice. Hiding a button is not access control.
- **I7 — Tenant queries are scoped.** Every Kysely query against agents / users / teams MUST include a `where('owner_id', '=', subject.userId)` (or team-membership equivalent). Bare `SELECT * FROM …_agents` MUST fail review.
- **I8 — Untrusted prompts are visible.** A team agent's system_prompt is user-authored content executed on another user's behalf — display it before chat, warn on changes (deferred to Week 10-12 UI; flag-friendly hook payload now).
- **I9 — Tokens never leak through hook return values.** Cookies set via `Set-Cookie` headers; bearer tokens never appear in `agents:resolve`, `auth:require-user`, or `chat:run` response payloads. Mirrors session-postgres I9 from Week 4-6.
- **I10 — Session ↔ agent immutability.** Once a session has an `agent_id`, it cannot change. Switching agents = new session. Simplifies authz caching and tool-catalog invariance.
- **I11 — IdP discovery is pinned.** OIDC discovery URL is part of plugin config; we do not auto-fetch from a user-supplied issuer at runtime without a config-time allowlist.
- **I12 — Bootstrap is one-shot.** `ax-next admin bootstrap` creates the first admin AND records that bootstrap is done. Running it twice on a populated DB is a no-op (idempotent) but emits a loud log.

These are the recommendations from `2026-04-24-week-9.5-multi-tenant-handoff.md` §"Scope decisions" applied as-written. Confirm with the user before starting Task 1 if any need to change:

7. **Per-agent MCP scoping:** Enforced in `@ax/tool-dispatcher`, not `@ax/mcp-client`.

// Service hook (this plugin registers exactly one impl).
'http:register-route' (ctx, {
  method: 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE',
  path: string,                    // exact match, no params yet — keep YAGNI
  handler: HttpRouteHandler,       // (req, res, scope) => Promise<void>

**Boundary review answers (paste into commit message):**
- Alternate impl: a future `http-server-fastify` could replace this; both register `http:register-route` with the same opaque handler signature.
- Field names: `method` / `path` / `handler` are HTTP-vocab but HTTP IS the alternate-impl set; `Set-Cookie` etc. live behind `res`, an opaque adapter. `req` exposes only `headers`, `body`, `cookies` accessors — never the raw `IncomingMessage`. Subscribers cannot reach the underlying socket.
- Subscriber risk: the request subscriber gets `headers` as `Record<string, string>` (already stringified, lowercased keys) — no buffer / stream leak.
- Wire surface: NOT an IPC action. Routes are host-internal; agents never call `http:register-route`.

## Task 4: Implement OIDC + dev-bootstrap auth flow in `@ax/auth` [security]

**Goal:** Wire the OAuth/OIDC handshake + cookie session. Two providers selected via config: `google-oidc` (real) and `dev-bootstrap` (env-token only, refused outside `NODE_ENV !== 'production'`).

**Files:**
- Modify: `packages/auth/src/plugin.ts`
- Create: `packages/auth/src/oidc.ts`
- Create: `packages/auth/src/dev-bootstrap.ts`
- Create: `packages/auth/src/store.ts`
- Create: `packages/auth/src/__tests__/oidc.test.ts`
- Create: `packages/auth/src/__tests__/dev-bootstrap.test.ts`
- Create: `packages/auth/src/__tests__/cookie-session.test.ts`

**Routes registered via `http:register-route`:**
- `GET /auth/sign-in/:provider` → 302 to IdP (`provider=google` only for MVP)
- `GET /auth/callback/:provider` → exchange code, mint session cookie, 302 to `/`
- `POST /auth/sign-out` → clear cookie + delete `auth_v1_sessions` row
- `POST /auth/dev-bootstrap` (only when `NODE_ENV !== 'production'` AND `AX_DEV_BOOTSTRAP_TOKEN` env is set AND request body's `token` matches) → mint bootstrap user + cookie

**OIDC flow rules:**
- State + PKCE (nonce too — `openid-client` handles all three).
- State stored in a temporary signed cookie (5-min lifetime), verified on callback.
- Redirect URI MUST be exact-matched against `config.redirectUris`. No wildcards.
- On callback failure, log `auth_callback_failed` with `provider + error.code` (NEVER the raw error from the IdP — may contain user-controlled `state`).

**Rate limit:** `/auth/sign-in/*` and `/auth/callback/*` capped at 30/min/IP via in-memory token bucket (good enough for MVP; multi-replica handles via `eventbus-postgres` in Week 13+).

**`auth:require-user` impl:**
1. Read signed `ax_auth_session` cookie.
2. Validate signature; on tamper → reject `unauthenticated`.
3. SELECT user_id FROM auth_v1_sessions WHERE session_id=$1 AND expires_at > NOW().
4. If none → reject `unauthenticated`.
5. SELECT * FROM auth_v1_users WHERE user_id=$1 → return User.

## Task 9: Admin agent endpoints in `@ax/agents` [security]

**Goal:** Mount `POST/GET/PATCH/DELETE /admin/agents[/:id]` on `@ax/http-server` with auth + ACL.

**Files:**
- Modify: `packages/agents/src/plugin.ts`
- Create: `packages/agents/src/admin-routes.ts`
- Create: `packages/agents/src/__tests__/admin-routes.test.ts`

**Pattern:**

```ts
async init({ bus }) {
  // ...
  await bus.call('http:register-route', ctx, {
    method: 'POST', path: '/admin/agents',
    handler: async (req, res) => {
      const auth = await bus.call('auth:require-user', ctx, { req });
      if (auth.rejected) return res.status(401).json({ error: 'unauthenticated' });
      const body = await req.json();
      const validated = AgentInputSchema.parse(body);   // zod
      const agent = await bus.call('agents:create', ctx, {
        actor: auth.user, input: validated,
      });
      res.status(201).json(toJson(agent));
    },
  });
  // ... GET, PATCH, DELETE
}
```

**Authz rules per endpoint:**
- `POST /admin/agents` — any authenticated user. Sets `owner_id = actor.id`, `owner_type = 'user'`. Team agents require `actor.isAdmin` OR membership of the named team (deferred to teams plugin in Task 13).
- `GET /admin/agents` — returns `agents:list-for-user(actor.id)` (already scoped).
- `PATCH /admin/agents/:id` — caller must be the agent's owner OR (for team agents) team admin.
- `DELETE /admin/agents/:id` — same.

**Input validation (zod):** `display_name` 1-128 chars, `system_prompt` ≤ 32 KiB, `allowed_tools` ≤ 100 entries each ≤ 64 chars, `mcp_config_ids` ≤ 50 entries, `model` from a hardcoded allow-list (`AX_AGENT_MODELS_ALLOWED` env, defaults to the 3 Claude IDs from CLAUDE.md).

**Tests:**
- Anonymous → 401.
- User A creates agent; User B GET /admin/agents → empty.
- User B PATCH User A's agent → 403.
- Body too large (> 1 MiB) → 413.

**Step 1-4: TDD pass for each endpoint, commit per logical group (POST, GET, PATCH, DELETE).**

+ agents:resolve. Input bounded by zod; model in allow-list.
