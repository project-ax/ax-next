# Week 6.5a — PR notes (scratch, lifted into PR body at push time)

## Summary

- Moves the agent turn-loop from in-process (`packages/core/src/chat-loop.ts`) to a subprocess sandbox driven via HTTP+JSON IPC over a per-session unix socket.
- Adds eight new packages: `@ax/ipc-protocol`, `@ax/ipc-server`, `@ax/session-inmemory`, `@ax/chat-orchestrator`, `@ax/agent-runner-core`, `@ax/agent-native-runner`, `@ax/tool-bash-impl`, `@ax/tool-file-io-impl`.
- Extends `@ax/sandbox-subprocess` with `sandbox:open-session`; deletes `sandbox:spawn` + `@ax/core`'s `SandboxSpawn*` + `@ax/core`'s `chat-loop.ts`.
- Refactors tool registration to `tool:register` / `tool:list` on `@ax/tool-dispatcher`; tool plugins become descriptor-only.
- Preserves Week 4–6 chat behavior; new acceptance test asserts bash runs as a grandchild of the host (proves the topology shift took).

## Security review — `@ax/ipc-server`

- Sandbox: New listening unix socket. Bound to a per-session private tempdir (mode `0700`, created via `fs.mkdtemp`), so only the host user can connect. Socket path never logged above `debug`. Every request requires a bearer token matching the session store; unknown tokens return `401` without consuming the request body. Body size capped at 4 MiB (matches `@ax/core`'s `MAX_FRAME`, invariant I11); requests exceeding return `413` before the body is fully read. `Content-Type: application/json` required on POST; other types return `415`. Only POST (RPCs) and GET (long-poll) are accepted; everything else is `405`. Long-poll timeout capped server-side at 30 s (I12); client can't pin a request open indefinitely. Server's idle socket timeout is set to 60 s so long-polls aren't killed mid-flight.
- Injection: Every inbound payload is Zod-parsed via `@ax/ipc-protocol` before any subscriber or service fires. Tool output and LLM output crossing back over IPC are treated as untrusted strings (per ax-conventions I5); they reach `chat:end` subscribers unchanged, never interpolated into another subprocess or SQL. Auth-token values are never echoed in error responses — the auth middleware's 401 body carries a generic `SESSION_INVALID` envelope without the offending token. Request bodies are read into a capped Buffer; no streaming into shell / filesystem / DB.
- Supply chain: No new runtime dependencies. Node built-ins `http`, `net`, `fs`, `crypto`, `os`, `path` only. `zod` already a transitive dep via `@ax/core` and directly via `@ax/ipc-protocol`. No postinstall scripts in this package. pnpm-lock diff: zero new entries.

## Security review — `@ax/sandbox-subprocess` session-lifetime extension

- Sandbox: Spawns the runner binary via the same hardened `spawn()` shape we already ship (`shell: false`, fixed argv `node <runnerBinary>`, `runnerBinary` validated to be absolute and to exist + be readable before spawn). Env injected: ONLY `AX_IPC_SOCKET`, `AX_SESSION_ID`, `AX_AUTH_TOKEN`, `AX_WORKSPACE_ROOT`, plus the existing parent-merged allowlist (`PATH`, `HOME`, `LANG`, `LC_ALL`, `TZ`, `NODE_OPTIONS=''`). Caller env is filtered to the allowlist. Auth token never echoed in error messages, never logged at `info` (I9). Unix socket and its parent dir created via `fs.mkdtemp` (mode 0700, I10) so only the host user can connect — the child inherits access because it runs as the same uid, not because the socket is world-readable. On child close / kill, `session:terminate` fires before the handle resolves so a post-exit IPC attempt from a stale token returns 401, not a leaked session. `ipc:stop` + tempdir `rm -r` also run on close — best-effort; failures log at warn and don't block the close.
- Injection: Runner gets user input via `session:claim-work` — which is in-process, not a shell interpolation. The auth token is crypto-random; runner never originates it. No caller-provided string reaches `spawn()`'s argv beyond the `runnerBinary` path, which is fixed by config at boot, not per-call.
- Supply chain: No new runtime deps. `@ax/ipc-protocol` is a sibling workspace package with its own review. No postinstall scripts added.

## Boundary review

### `session:*` family (`@ax/session-inmemory`)

- **Alternate impl:** `@ax/session-postgres` in Week 7–9 — identical contract, pg-backed inbox with `LISTEN/NOTIFY` wakeup instead of in-memory waiters.
- **Payload fields:** `sessionId`, `token`, `cursor`, `workspaceRoot`, `entry.type`, `entry.payload`. No sqlite / pg vocabulary. `cursor` is a monotonic int — identical semantics in both backends. `token` is opaque base64url, not JWT.
- **Subscriber risk:** these are all service hooks; no subscribers.
- **Wire surface:** none exposed via IPC directly. `session:claim-work` is reached transitively through `GET /session.next-message` in `@ax/ipc-protocol`, whose schema is owned by the IPC protocol package per design-doc "Schema location."

### `sandbox:open-session` (`@ax/sandbox-subprocess`)

- **Alternate impl:** `@ax/sandbox-k8s` in Week 7–9 — pod-per-session. `handle.kill()` SIGTERMs the pod; `socketPath` becomes a cluster-HTTP URL. Contract identical, field `socketPath` generic enough to carry either transport.
- **Payload fields:** `sessionId`, `workspaceRoot`, `runnerBinary`, `socketPath`. No docker / kubernetes vocabulary (no `imageRef`, `podName`, `namespace`).
- **Subscriber risk:** service hook, no subscribers.
- **Wire surface:** not exposed directly; callable only in-process.

### `tool:register` / `tool:list` (`@ax/tool-dispatcher`)

- **Alternate impl:** none plausible today — the catalog is in-memory host state. If a future multi-replica host needs a shared catalog, it'd register `tool:list` from a postgres-backed plugin. The current shape allows that.
- **Payload fields:** `ToolDescriptor { name, description, inputSchema, executesIn }`. `executesIn` is a closed enum (`sandbox` | `host`) — not a backend handle. `inputSchema` is an opaque JSON-Schema object (same pattern as Week 4–6).
- **Subscriber risk:** service hooks, no subscribers.
- **Wire surface:** `tool.list` IPC action — schema in `@ax/ipc-protocol`.

### IPC protocol (`@ax/ipc-protocol`)

- **Alternate impl:** transport is swappable per sandbox plugin (unix socket vs cluster HTTP); protocol schemas are identical. See design doc §"IPC protocol".
- **Payload fields:** `messages`, `tools`, `call`, `cursor`, `version`, `delta`. `version` is a `WorkspaceVersion` branded opaque token. No git / pg / k8s vocabulary.
- **Subscriber risk:** the hooks fired on IPC receipt (`llm:pre-call`, `tool:pre-call`, `tool:post-call`, `chat:end`, `chat:turn-end`) already have subscribers from Week 4–6 (audit-log) — their payload shapes are unchanged, so no subscriber breakage.
- **Wire surface:** this IS the wire surface. Every action schema is versioned implicitly by the `@ax/ipc-protocol` package version; 6.5b extends it additively (streaming variants), so no break-the-wire changes this slice.

## Deviations from the plan

Recorded so a reviewer doesn't need to cross-check.

1. **Orchestrator does NOT call `session:create`.** The plan sketched it as step 2 of the per-chat flow, but `@ax/sandbox-subprocess`'s `sandbox:open-session` already mints the session + token internally (so the token flows only into the runner's env — I9). A second `session:create` throws `duplicate-session`. Orchestrator delegates session minting to the sandbox plugin; its `calls` manifest reflects this (no `session:create`).
2. **One-shot auto-cancel in the orchestrator.** The design keeps the runner persistent (inbox loops forever until cancel). The CLI is one-shot. Without explicit bridging, the CLI's `chat:run` would hit its 10-minute timeout. Fix: orchestrator subscribes to `chat:turn-end` and, in one-shot mode (default for 6.5a), queues a `cancel` into the inbox after the first turn. Multi-message callers set `oneShot: false`.
3. **`tool.execute-host` handler is wired but has no registered tools in 6.5a.** It returns 404 VALIDATION when dispatched; the 404 path is tested. Happy path lands in Week 10+ when `@ax/memory-cortex` or similar host-side tool plugins arrive. Keeping the handler wired (rather than 501) lets the IPC server's route table stay uniform with Task 4's dispatcher and exercises the negative path.
4. **`workspace.commit-notify` is a stub** — returns `{accepted: true, version: 'stub', delta: null}` unconditionally; no `workspace:pre-apply` / `workspace:applied` subscribers fire. Real workspace plugins land in Week 7–9 per plan.
5. **`@ax/core`'s `PluginErrorCode` widened with `| (string & {})`.** Caught during Task 2's review — the closed union forced `as PluginErrorCode` casts for any new plugin that wanted domain-specific codes. The widening preserves autocomplete on the 11 known codes and removes the cast.
