# Week 6.5d — claude-sdk runner Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `@ax/agent-claude-sdk-runner` (sandbox-side) + `@ax/llm-proxy-anthropic-format` (host-side) so AX can swap the sandbox runner to `@anthropic-ai/claude-agent-sdk` via config, with claude-sdk's built-in tools AND host-mediated tools both firing the existing `tool:pre-call` / `tool:post-call` subscribers through the unchanged IPC contract.

**Architecture:**

- The runner wraps `query()` from `@anthropic-ai/claude-agent-sdk`. Built-in tools (`Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `TodoWrite`) stay enabled and run inside the sandbox process tree (claude-sdk spawns subprocess descendants); `WebFetch`, `WebSearch`, `Skill`, `Task` are disabled via `disallowedTools`. Any `executesIn: 'host'` tool from `tool.list` is surfaced to the SDK through an in-process `createSdkMcpServer` whose handlers forward to `tool.execute-host` IPC.
- `canUseTool` callback issues `tool.pre-call` IPC (veto + rewrite); `hooks.PostToolUse` issues `event.tool-post-call` IPC (observation-only per D4). Works uniformly for built-ins and SDK-MCP tools.
- LLM traffic routes via `ANTHROPIC_BASE_URL` → host-side `@ax/llm-proxy-anthropic-format` → `llm:call` service hook. The proxy translates Anthropic Messages API wire format (`POST /v1/messages` with multi-block content, `tool_use`/`tool_result`, system string, tools array, streaming SSE) into `@ax/core`'s flat `LlmRequest` shape and back. Session bearer token is reused as `ANTHROPIC_API_KEY`; the proxy verifies it via `session:resolve-token`.
- No cross-plugin imports (I2). Schema lives in `@ax/ipc-protocol` (unchanged — 6.5d adds **no** new IPC actions; host-mediated tools reuse `tool.execute-host`).

**Tech Stack:** TypeScript, pnpm workspace, Zod schemas, Node `http` (proxy listener), `@anthropic-ai/claude-agent-sdk` (sandbox-side), vitest, existing `@ax/agent-runner-core` + `@ax/ipc-protocol` unchanged.

**Branch:** `feat/week-6.5d-claude-sdk-runner` off `main` (tip of 6.5a, which is `f07a6cf`). The kickoff prompt said "branch off 6.5c," but 6.5b/6.5c are skipped for MVP per `project_mvp_direction.md`. Flag this to reviewer in the PR description.

## Prior-slice deviations folded into invariants

6.5a (PR #6) surfaced the following lessons; 6.5d carries them forward as numbered invariants the reviewer can audit against.

- **I1 — runner holds no LLM credentials.** The sandbox never sees `ANTHROPIC_API_KEY` in the real sense. The env var passed to claude-sdk is the per-session bearer token (same token that authenticates IPC). The proxy verifies it via `session:resolve-token`, same path `@ax/ipc-server` uses.
- **I2 — schema source of truth stays in `@ax/ipc-protocol`.** 6.5d adds NO new IPC actions. `tool.execute-host` (already defined, schema-only in 6.5a) earns its runtime wiring here. If a field needs adding, add it in `@ax/ipc-protocol` in one task and consume it in the next.
- **I3 — proxy HTTP surface is a new attack surface.** Bind to `127.0.0.1` only, emit `Authorization: Bearer` check BEFORE any body parsing, enforce body-size cap identical to `@ax/ipc-server` (4 MiB), redact token in logs. Security-checklist below is mandatory.
- **I4 — no half-wired plugins.** Both packages are wired through to the acceptance test in the same PR. The runner binary is resolvable from `@ax/cli` via `createRequire` (same mechanism as `@ax/agent-native-runner`).
- **I5 — honest tool name passthrough.** The claude-sdk tool name (`Bash`, `Read`, …) is what flows to `tool:pre-call` subscribers. Do NOT remap to `bash` / `read_file` — that would forge the audit trail, and pi-session runner (Week 13+) would need its own remap. Subscribers that care about a specific tool handle the naming difference themselves.
- **I6 — no new dependency on `@ax/tool-bash-impl` / `@ax/tool-file-io-impl`.** claude-sdk ships its own Bash/Read/Write/Edit. The sandbox-side impl packages stay wired to `@ax/agent-native-runner` only; the claude-sdk runner does not import them.
- **I7 — turn boundary = `SDKResultMessage`.** `event.turn-end` and `event.chat-end` fire on the SDK's terminal result, never mid-stream. Workspace commits (when Week 7–9 ships them) happen here; do not emit them per-tool.
- **I8 — streaming SSE from the proxy is synthesized, not pass-through.** IPC `llm.call` is non-streaming. The proxy issues one IPC call per `/v1/messages` request and then emits faux SSE frames (`message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`) to satisfy claude-code CLI's streaming expectations. No partial-token fan-out in 6.5d; 6.5b-style streaming is Week 13+.
- **I9 — proxy listener lifecycle mirrors IPC listener.** Per-session bind (ephemeral port on 127.0.0.1), started alongside `ipc:start` and torn down alongside `ipc:stop`. Host-wide singleton is a k8s concern (Week 7–9), not MVP.

## Scope decisions this plan locks in

Surfaced by the design doc for 6.5d; each needs an explicit call before code lands.

1. **Streaming vs non-streaming proxy.** Support BOTH. claude-code CLI defaults to `stream: true`; if we ship non-streaming only we lose the built-in. Non-streaming path: direct bulk JSON response translated from one `llm:call`. Streaming path: one `llm:call` IPC, synthesize SSE frames from the bulk response. Token-level streaming from the provider is NOT re-streamed (that's 6.5b work we skipped).

2. **Tool name convention.** Pass the claude-sdk tool name verbatim to `tool:pre-call` and `tool:post-call`. MCP tools reaching claude-sdk via `createSdkMcpServer` carry the `mcp__ax-host-tools__<toolname>` prefix the SDK synthesizes; our `tool:pre-call` subscriber receives this prefixed name. The `executeToolHost` IPC dispatch inside the runner strips the prefix before calling `tool.execute-host` (host expects the bare tool name).

3. **MCP server grouping.** One `createSdkMcpServer({ name: 'ax-host-tools', tools: [...] })` per runner. All `executesIn: 'host'` tools from `tool.list` register into this one server. Keeps the tool namespace flat and the SDK surface predictable.

4. **Disabled built-ins.** `disallowedTools: ['WebFetch', 'WebSearch', 'Skill', 'Task']`. Do NOT use `allowedTools` (whitelist) — that would require re-enumerating every SDK-MCP tool and is fragile when the SDK adds new built-ins.

5. **Settings sources.** `settingSources: []` (claude-sdk-docs default). We do NOT want CLAUDE.md / project settings bleeding into the sandbox session. AX owns the context.

6. **System prompt.** `systemPrompt: { type: 'preset', preset: 'claude_code' }`. The runner's job is to be claude-sdk; reinventing the system prompt here would defeat the point. Future: allow CLI config to append.

7. **`pathToClaudeCodeExecutable`.** Leave unset. The SDK bundles its own claude-code CLI. Documented side effect: the sandbox `node_modules` must contain `@anthropic-ai/claude-agent-sdk` at runtime — covered by the runner package's dependency and pnpm's hoisting.

8. **Proxy auth scheme.** Reuse the existing session bearer token. No new auth boundary. The proxy calls `session:resolve-token` to validate — same check `@ax/ipc-server` does.

9. **Proxy transport.** Plain HTTP on `127.0.0.1:<ephemeral>` per session, URL injected via `ANTHROPIC_BASE_URL=http://127.0.0.1:<port>`. The sandbox subprocess inherits from the host's network namespace so this works. k8s / Docker sandbox plugins will have to reintroduce routing (Week 7–9); the plan documents this as a known extension point.

10. **Built-in vs MCP `tool:pre-call` parity.** Per design doc D4 + the acceptance test: every tool call — built-in or MCP — must fire `tool:pre-call` with a `ToolCall` that has `{id, name, input}` and respect the verdict. `canUseTool` gives us `toolName` + `input` only (no tool_use_id), so we synthesize a runner-local uuid for the IPC call. PostToolUse gets the real tool_use_id; we emit that as the `id` on `event.tool-post-call`. Subscribers correlating pre↔post by id should not (per D4, post is observation-only); we do not expose a correlation contract in 6.5d.

### Task 10: Wire `query()` — runner main entry point

**Files:**
- Modify: `packages/agent-claude-sdk-runner/src/main.ts`

**Step 1: Write main body.**

```ts
import { query } from '@anthropic-ai/claude-agent-sdk';
import {
  createIpcClient,
  createInboxLoop,
} from '@ax/agent-runner-core';
import type {
  ChatMessage,
  ToolListResponse,
  ToolDescriptor,
} from '@ax/ipc-protocol';
import { readRunnerEnv } from './env.js';
import { createCanUseTool } from './can-use-tool.js';
import { createPostToolUseHook } from './post-tool-use.js';
import { createHostMcpServer } from './host-mcp-server.js';

const DISABLED_BUILTINS = ['WebFetch', 'WebSearch', 'Skill', 'Task'] as const;

export async function main(): Promise<number> {
  const env = readRunnerEnv();
  const client = createIpcClient({
    socketPath: env.ipcSocket,
    token: env.authToken,
  });

  const { tools } = (await client.call('tool.list', {})) as ToolListResponse;
  const hostMcpServer = createHostMcpServer({ client, tools });

  const inbox = createInboxLoop({ client });
  const history: ChatMessage[] = [];

  try {
    for (;;) {
      const entry = await inbox.next();
      if (entry.type === 'cancel') break;
      if (entry.payload === undefined) continue;

      const userText = entry.payload.content;

      // claude-sdk's `query()` accepts string OR async iterable. We use a
      // string per call since AX's inbox model is one-user-message-per-turn.
      // `continue: true` on subsequent calls reuses the prior session.
      const queryIter = query({
        prompt: userText,
        options: {
          env: {
            ANTHROPIC_BASE_URL: env.llmProxyUrl,
            ANTHROPIC_API_KEY: env.authToken,
          },
          cwd: env.workspaceRoot,
          disallowedTools: [...DISABLED_BUILTINS],
          canUseTool: createCanUseTool({ client }),
          hooks: {
            PostToolUse: [
              { hooks: [createPostToolUseHook({ client })] },
            ],
          },
          mcpServers: { 'ax-host-tools': hostMcpServer },
          settingSources: [],
          systemPrompt: { type: 'preset', preset: 'claude_code' },
          continue: history.length > 0,
        },
      });

      for await (const msg of queryIter) {
        if (msg.type === 'assistant') {
          // Text blocks accumulate into an assistant ChatMessage for history.
          const text = msg.message.content
            .filter((b) => b.type === 'text')
            .map((b) => (b as { text: string }).text)
            .join('\n');
          if (text) history.push({ role: 'assistant', content: text });
        } else if (msg.type === 'result') {
          // Terminal. I7: turn-end fires here, not per-tool.
          await client
            .event('event.turn-end', { reason: 'user-message-wait' })
            .catch(() => {});
          break;
        }
      }
    }

    await client
      .event('event.chat-end', {
        outcome: { kind: 'complete', messages: history },
      })
      .catch(() => {});
    await client.close();
    return 0;
  } catch (err) {
    await client
      .event('event.chat-end', {
        outcome: {
          kind: 'terminated',
          reason: err instanceof Error ? err.name : 'unknown',
          error: err,
        },
      })
      .catch(() => {});
    await client.close();
    return 1;
  }
}
```

### Task 15: Security checklist — `@ax/llm-proxy-anthropic-format` + claude-sdk deps

**REQUIRED SUB-SKILL: Invoke `security-checklist` skill twice — once for the proxy HTTP surface, once for the `@anthropic-ai/claude-agent-sdk` dependency tree.**

**Files:**

- Create: `packages/agent-claude-sdk-runner/SECURITY.md`
- Modify: PR description per the skill's output.

**Step 1: Run `security-checklist` skill for the proxy.**

Three threat models:

1. **Sandbox escape:** the proxy is HOST-SIDE, listening on `127.0.0.1`. Sandbox subprocess reaches it via the loopback interface. Risks: if the proxy's auth is bypassed, ANY local process with `127.0.0.1` reach could ask the host to issue LLM calls on behalf of a session. Mitigation: session bearer token check (`session:resolve-token`) before body read; per-session ephemeral port; body-size cap. Document in SECURITY.md.
2. **Prompt injection:** the proxy translates untrusted model output back-and-forth (tool results → user messages → LLM → assistant → proxy → claude-sdk). The `llm:pre-call` subscribers on the host side (canary, etc., when they arrive) intercept. The proxy does NOT need its own scrubbing — it's a translator, not a firewall. Document explicitly to prevent a future reviewer from adding scrubbing in the wrong layer.
3. **Supply chain:** `@anthropic-ai/claude-agent-sdk` is a new top-level dep. Pin to an exact patch version. Document how to update (pnpm add with exact version, run acceptance test, security-checklist re-run).

**Step 2: Run `security-checklist` skill for the claude-sdk dependency tree.**

- The SDK bundles the claude-code CLI (a subprocess binary); its own deps land transitively. Spot-check the pnpm-lock diff for unexpected additions.
- Built-in tools that remain enabled (`Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `TodoWrite`) all operate inside the sandbox subprocess tree. Verify each runs with cwd = workspaceRoot; `Bash` specifically inherits the sandbox's filesystem scope.
