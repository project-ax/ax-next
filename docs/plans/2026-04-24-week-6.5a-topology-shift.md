# Week 6.5a — Topology Shift Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to execute this
> plan task-by-task in the current session. Each task dispatches a fresh implementer subagent,
> then a spec-compliance reviewer, then a code-quality reviewer (see the skill for the two-stage
> review loop).

**Goal:** Move the chat loop out of the host process and into a subprocess sandbox with a clean
IPC contract, while preserving the exact observable behavior of Week 4–6's chat end-to-end.

**Architecture:** Host keeps orchestration, untrusted-content subscribers, tool catalog, and LLM
credentials. Sandbox runs the agent turn-loop and executes sandbox-local tools (bash, file I/O).
All host↔sandbox traffic is HTTP+JSON over a per-session unix socket, sandbox-initiated, with a
bearer token minted at session-open time. No streaming, no new providers, no workspace backend —
those land in 6.5b/7–9.

**Tech Stack:** TypeScript, Node ≥ 20 (built-in `http` + `net`), pnpm workspaces, Zod for wire
schemas, Vitest. No new runtime dependencies beyond what Week 4–6 already pins.

**Contract doc:** `docs/plans/2026-04-24-week-6.5-agent-sandbox-design.md` — read it before touching
any task; section references below are to this doc.

**Branch:** `feat/week-6.5a-topology-shift`, off `main` (Week 4–6 shipped as PR #5 → merged).

## Invariants for this slice (fold reviewer gaps in as we hit them)

Numbered so later tasks can point to them. Every PR iteration that uncovers a new invariant gets
a numbered entry appended here.

- **I1 — Observable chat behavior preserved.** The Week 4–6 library-mode E2E test (Anthropic
  stubbed, real bash via sandbox, audit-log persisted) must pass unchanged after 6.5a, minus the
  fact that the turn-loop now runs in a child process.
- **I2 — No half-wired packages.** Every new package has a runtime caller reaching it on the
  acceptance-test path before it merges (ax-conventions I3).
- **I3 — Shared-import allowlist grows by exactly two.** `@ax/ipc-protocol` and
  `@ax/agent-runner-core` get added to the ESLint `no-restricted-imports` allowlist; nothing
  else. CLI and presets remain the only arbitrary-import exception.
- **I4 — No leaky hook-payload field names.** Session / IPC / tool-catalog hooks use generic
  names (`cursor` not `rowid`, `version` not `sha`, `token` not `jwt`). Boundary-review block in
  the PR description justifies every payload field.
- **I5 — Capabilities minimized on both new trust boundaries.** `@ax/ipc-server` and
  `@ax/sandbox-subprocess`'s session-lifetime extension each ship with a `SECURITY.md` walking
  the three threat models. Re-run the `security-checklist` skill for each.
- **I6 — `sandbox:spawn` is deleted in the same PR that adds `sandbox:open-session`.** No dead
  code, no "we'll delete it later," no feature flag (scope decision 7 of the design doc).
- **I7 — `chat-loop.ts` is deleted in the same PR that registers `chat:run` from
  `@ax/chat-orchestrator`.** Same no-dead-code rule.
- **I8 — Runner binary path is explicit.** `sandbox:open-session` input takes an absolute
  `runnerBinary` path; the CLI resolves it from the runner package at boot. No env-var fallback,
  no implicit `process.execPath` magic — a caller that forgot to configure it gets a loud error.
- **I9 — Auth tokens are opaque crypto random, never JWT.** 32 bytes from `crypto.randomBytes`,
  base64url-encoded (no padding). Never logged at info level; redact in any error path that
  might echo the request.
- **I10 — Unix socket lives in a private tempdir.** `fs.mkdtemp` under `os.tmpdir()` with mode
  `0700`; socket file inherits the dir's perms. No world-readable sockets, no shared names.
- **I11 — IPC body-size cap at the listener.** 4 MiB (matches `@ax/core`'s `MAX_FRAME` already
  shipped in Week 4–6). Requests exceeding the cap get rejected with HTTP 413 before the body is
  fully read.
- **I12 — Long-poll timeout is bounded.** `session.next-message` caps at 30s server-side; the
  client (runner) reconnects on timeout with exponential backoff capped at ~30s (design doc
  "Error model").

**Hooks this plugin registers** (all service hooks; exactly-one-producer):

- `session:create` — input `{ sessionId?: string, workspaceRoot: string }`, output
  `{ sessionId, token }`. Mints a session record + fresh opaque token. `sessionId` optional so
  the orchestrator can pass its own (ChatContext-derived) ID.
- `session:resolve-token` — input `{ token: string }`, output `{ sessionId, workspaceRoot } |
  null`. Used by `@ax/ipc-server`'s auth middleware. Returns `null` on revoked / unknown.
- `session:queue-work` — input `{ sessionId, entry: InboxEntry }`, output `{ cursor: number }`.
  `InboxEntry = { type: 'user-message', payload: ChatMessage } | { type: 'cancel' }`.
- `session:claim-work` — input `{ sessionId, cursor: number, timeoutMs: number }`, output
  `{ type: 'user-message', payload, cursor } | { type: 'cancel', cursor } | { type: 'timeout',
  cursor }`. Long-poll: resolves immediately if a later entry exists, otherwise waits up to
  `timeoutMs` for a new entry.
- `session:terminate` — input `{ sessionId }`, output `{}`. Marks session terminated; token
  lookups return `null` thereafter; any waiting `session:claim-work` promises resolve with
  `timeout` (they will see a terminated session on their next resolve attempt and the IPC server
  will translate that into `SESSION_INVALID`).

   known path... wait. Per-session socket path means the server binds dynamically. Revise:
   - `sandbox:open-session` returns the `socketPath`. The IPC server's `ipc:start` takes the
     same socketPath and binds it. So ordering is:
     a. `session:create` → token.
     b. Generate socket path.
     c. `ipc:start({ socketPath })`.
     d. Spawn runner with env — runner connects to socketPath.
     e. `session:queue-work` with the user message.
   - `@ax/sandbox-subprocess`'s `sandbox:open-session` composes (b)+(d) and calls `ipc:start` in
     between. It declares `calls: ['ipc:start', …]` accordingly. Orchestrator just calls
     `sandbox:open-session`.
5. Orchestrator registers a per-session deferred (keyed by `ctx.sessionId`) in a Map. Its
   `chat:end` subscriber checks the sessionId and resolves the deferred with the outcome.
6. Orchestrator awaits the deferred (with a bounded chat-level timeout, configurable, default
   10 min — matches v1's safety net).
7. Cleanup: `handle.kill()` + `ipc:stop` + remove from the deferred map. `session:terminate` is
   already fired by `sandbox:open-session`'s on-close hook in Task 5.
8. Return the outcome.
