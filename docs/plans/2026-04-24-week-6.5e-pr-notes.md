# Week 6.5e ‚Äî PR notes (scratch, lifted into PR body at push time)

## Summary

Week 6.5e ships two new plugins: `@ax/credentials` (at-rest encrypted secrets, AES-256-GCM via `node:crypto`, key sourced from `AX_CREDENTIALS_KEY`) and `@ax/mcp-client` (host-side Model Context Protocol server hosting ‚Äî stdio + streamable-http + SSE transports, per-server connection lifecycle with reconnect backoff). The canary is `packages/cli/src/__tests__/mcp-client.e2e.test.ts` ‚Äî it spawns the real `@ax/mcp-client` against a real stdio MCP subprocess (`mcp-server-stub` from `@ax/test-harness`), calls a tool successfully, crashes the server, and verifies the recovery path surfaces `MCP_SERVER_UNAVAILABLE` without terminating chat. Runs in ~854ms locally.

## Scope

What shipped:

- **`@ax/credentials`** ‚Äî three service hooks (`credentials:set`, `credentials:get`, `credentials:delete`), AES-256-GCM round-trip through `storage:set` / `storage:get`, 32-byte key read once from `AX_CREDENTIALS_KEY` at plugin init (hex or base64 accepted), ID validation `^[a-z0-9][a-z0-9_.-]{0,127}$`, tombstone delete (encrypted empty string) until `storage:delete` lands. Host-side only ‚Äî not exposed on the IPC bridge to sandboxes.
- **`@ax/mcp-client`** ‚Äî per-server `McpConnection` (`disconnected ‚Üí connecting ‚Üí ready` with `unhealthy` recoverable branch and `closed` one-way terminus); stdio / streamable-http / SSE transports via `@modelcontextprotocol/sdk@1.29.0` (exact-pinned to match the transitive resolution already pulled in by `@anthropic-ai/claude-agent-sdk@0.2.119`); tool-name namespacing (`mcp.<serverId>.<sanitized>`, with duplicate-remote-tool rejection and hash-suffix collision handling); reconnect backoff `1s, 2s, 4s, 8s, 16s` capped at 16s; `MCP_SERVER_UNAVAILABLE` wraps connection failures as tool-error results the model can reason about; plain `http://` URLs log `mcp_plain_http_transport` at startup.
- **`@ax/cli`** ‚Äî new `ax-next credentials set <id>` subcommand (stdin-only; no argv value to avoid shell history), `ax-next mcp add|list|rm|test` subcommands; both plugins conditionally registered on the chat path based on the user's `ax.config.ts`.
- **`@ax/test-harness`** ‚Äî `mcp-server-stub` (minimal stdio MCP server exposing `echo` and `crash` tools) for acceptance coverage.

What was deferred (target weeks in parens):

- Admin HTTP API for MCP server configs (Week 9.5 / 10‚Äì12) ‚Äî configs today are edited via CLI subcommands; no admin endpoint yet.
- Per-agent MCP scoping (Week 9.5) ‚Äî every MCP server is reachable from every agent in the current CLI process until the auth slice lands.
- `ax-next credentials rotate` subcommand (follow-up issue) ‚Äî today's rotation means restart + re-set each credential.
- `storage:delete` service hook (separate slice) ‚Äî `credentials:delete` is a tombstone write until then.
- OAuth flows for MCP servers requiring them (Week 13+) ‚Äî bearer tokens via `headerCredentialRefs` work today.
- Dynamic MCP config reload (Week 13+) ‚Äî `ax-next mcp add` requires a CLI restart to pick up.
- ulimits / cgroups for stdio MCP subprocesses (Week 13+) ‚Äî a badly-written community MCP server can OOM the host or spin CPU.
- TLS certificate pinning for HTTP / SSE MCP transports (Week 13+) ‚Äî trust the system CA bundle today.
- KMS-backed encryption key provider (Week 13+) ‚Äî `AX_CREDENTIALS_KEY` env-var is the MVP sentinel.
- Prefix-scan in `@ax/storage-sqlite` (separate slice) ‚Äî `ax-next mcp list` today reads a `mcp:servers` index key because `storage:get` is exact-key only.
- Token-level streaming from the MCP subprocess (Week 13+) ‚Äî tool results return bulk.

## Boundary review

This slice adds two new hook surfaces: `credentials:{set,get,delete}` and no new MCP-specific hooks (mcp-client consumes the existing `tool:register` / `tool:execute:${name}` / `storage:*` / `credentials:get` surfaces as a producer).

### `credentials:set` / `credentials:get` / `credentials:delete` (`@ax/credentials`)

- **Alternate impl this hook surface could have:** `@ax/credentials-kms` ‚Äî same three service hooks, but backed by a cloud KMS decrypt call instead of local AES-GCM. The service hook signatures are plaintext-in / plaintext-out to the caller, so swapping the implementation requires no change at the call site.
- **Payload field names that might leak:** none. The service hooks use `id` (opaque string) and `value` (opaque string). No mention of `key`, `iv`, `tag`, `cipher`, `kms-arn`, `rotation-id`, or any other backend-specific vocabulary on the hook surface. The `credential:` storage-key prefix is an internal implementation detail and does not appear in any hook payload.
- **Subscriber risk:** no subscribers today. A future `@ax/audit-log` subscriber keying off `id` is fine ‚Äî `id` is vendor-neutral. A subscriber that tried to peek at ciphertext would have to intercept `storage:set`, which is a different boundary.
- **Wire surface:** none. `credentials:*` is host-side only; it is NOT exposed on the IPC bridge to sandboxes. If we ever do expose it, the auth story (per-session scoping, per-credential ACL) has to land first.

### `@ax/mcp-client` ‚Äî no new hooks, new producer on existing ones

- **Alternate impl:** any other dynamic-tool-source plugin. An OpenAPI-to-tool bridge, an LLM-as-tool plugin, or a Home-Assistant-to-tool adapter would implement the same producer shape ‚Äî call `tool:register` per discovered tool at init, service `tool:execute:${name}` at call time.
- **Payload fields that might leak:** `transport` can be `'stdio' | 'streamable-http' | 'sse'`, which is closed MCP vocabulary (any MCP-speaking implementation uses the same three names), not backend-specific. `credentialRefs` / `headerCredentialRefs` values are opaque credential IDs, saying nothing about whether the backend is env-var / KMS / Vault.
- **Subscriber risk:** none new. Existing subscribers (dispatcher catalog, observers) already handle `executesIn: 'host'`.
- **Wire surface:** no new IPC actions. Sandboxes invoke MCP tools through the existing `tool.execute-host` IPC handler (schema-only in 6.5a, runtime-wired in 6.5d).

## Security review

Both plugins landed with full `security-checklist` SECURITY.md files. Condensed highlights:

- **`packages/credentials/SECURITY.md`** ‚Äî No new runtime deps (`node:crypto` + `@ax/core` only). AES-256-GCM is standard AEAD: 32-byte key, 12-byte random IV per encryption, 16-byte tag. Key is read once from env at plugin init, held in a closure for process lifetime; never logged, never returned over a hook. Decrypt failures return scrubbed `PluginError` that never echo ciphertext or key material. Credential IDs are validated before storage, then namespaced under `credential:` to prevent plugin-storage collisions. Known limits: no audit logging of reads (future subscriber), no per-credential ACL (Week 9.5 auth slice), no memory zeroization (V8-level concern, Week 13+ KMS), no ciphertext version byte (migration cost if we swap algorithms).
- **`packages/mcp-client/SECURITY.md`** ‚Äî One new runtime dep: `@modelcontextprotocol/sdk@1.29.0`, exact-pinned to align with the transitive resolution from `@anthropic-ai/claude-agent-sdk@0.2.119` (a caret range would double our audit surface). No `preinstall` / `postinstall` / `install` / `prepare` hooks in the SDK. Stdio subprocesses spawn with a short env allowlist (`PATH`, `HOME`, `LANG`, `LC_ALL`) plus `config.env` plus resolved `credentialRefs`; we explicitly pass `env` to `StdioClientTransport` so the SDK does NOT fall back to `getDefaultEnvironment()`. Inline-secret keys in the config (`password`, `secret`, `token`, `apikey`, `api_key` at any depth) are rejected at parse time. Argv is `{command, args}` with no shell interpolation. Tool descriptions / results are treated as opaque third-party content ‚Äî never interpolated into shells, paths, queries, or log lines. Known limits: no subprocess resource caps (Week 13+), no TLS cert pinning (Week 13+), no OAuth flows (Week 13+), no per-agent scoping (Week 9.5).

Noteworthy nuances documented for future readers:

- The acceptance test post-call output is double-wrapped via the IPC handler ‚Äî the outer envelope is the handler's structured response, the inner is MCP's tool-result shape. Future tool-observer plugins unwrapping this should do it explicitly.
- The `crash` tool race: the FIRST `callTool` invocation on a crash-tool also returns `MCP_SERVER_UNAVAILABLE`, because the subprocess exits during (not before) the JSON-RPC response. That's the path we want ‚Äî the plugin treats both flavors the same way ‚Äî but it's worth knowing the crash assertion is "you got an error," not "you got a specific flavor of error."

## Architecture / invariant audit

Which of the five invariants this slice exercised:

- **I1 (hook surface transport/storage-agnostic):** `credentials:*` uses `id` / `value`; no mention of `iv`, `tag`, `cipher`, `kms-arn`. `@ax/mcp-client` uses MCP-vocabulary closed names (`stdio | streamable-http | sse`), not backend-specific storage names. Boundary review passed.
- **I2 (no cross-plugin imports):** `@ax/mcp-client` talks to `@ax/credentials` only through the bus (`credentials:get`). `@ax/credentials` talks to `@ax/storage-sqlite` only through `storage:set` / `storage:get`. Verified by `eslint.config.mjs` `no-restricted-imports`.
- **I3 (no half-wired plugins):** Both plugins are reachable from the acceptance canary (`mcp-client.e2e.test.ts`) ‚Äî it invokes `credentials:set` to stash an API key, `@ax/mcp-client` resolves it via `credentials:get` at connect time.
- **I4 (one source of truth):** MCP server configs live exclusively in `@ax/storage-sqlite` via the `mcp:servers` index key + per-server config keys. `@ax/mcp-client` never caches configs outside what it needs for an active connection. Credentials live exclusively in `@ax/storage-sqlite` via the `credential:` prefix.
- **I5 (capabilities explicit and minimized):** Host-side plugins with no IPC exposure; stdio MCP subprocess env is an allowlist (not inherit-and-filter); inline secrets in config are rejected at parse time, not just warned. MCP subprocesses run host-side because granting the sandbox network egress + credential decryption would widen the trust boundary for no gain.
