# Week 10–12 — Web chat + conversation persistence PR notes

Closes the browser-chat MVP: users sign in, pick an agent, chat from the browser, and reload without losing context. Streaming + durable conversation history land on top of Week 9.5's tenant primitives.

## Summary

- Implementation plan: `docs/plans/2026-04-26-week-10-12-web-chat-conversations.md`.
- Handoff: `docs/plans/2026-04-23-week-10-12-handoff.md`.

## Security review rollup

### Sandbox / capability leakage

Net-new capabilities are minimal and bounded.

- **New IPC action `conversation.fetch-history`** — runner gains a bounded read of *its own* conversation's transcript. Authz: server-side handler resolves the runner's bearer token → `ctx.userId`; the bus-side `conversations:fetch-history` hook calls `conversations:get(conversationId, ctx.userId)` which `agents:resolve`-gates. A runner that forges a foreign `conversationId` still can't read it. Schema is `.strict({})` shape; no path / process / network widening.
- **New chat-flow HTTP routes** — `POST /api/chat/messages`, `GET /api/chat/conversations`, `GET /api/chat/conversations/:id`, `DELETE /api/chat/conversations/:id`, `GET /api/chat/agents`, `GET /api/chat/stream/:reqId` (SSE). All auth-gated via the existing `auth:require-user` cookie middleware. CSRF middleware (Week 9.5) inherited on POST/DELETE. ACL via `agents:resolve` at every entry that touches an agent-scoped row. Body cap: 1 MiB (http-server default); content-blocks further capped at 20 per message (`packages/channel-web/src/wire/chat.ts:57`).
- **SSE long-lived connections** — bounded by Node's socket-level timeouts and the chunk-buffer's 60-second idle TTL (`packages/channel-web/src/server/chunk-buffer.ts:33-34`). Per-request subscriber suffix prevents cross-client subscription leakage. `stream.onClose` cleanup unsubscribes both `chat:stream-chunk` and `chat:turn-end` filters.
- **No new FS / process-spawn / network reach** beyond what existed pre-10-12.

### Prompt injection / untrusted content

Trace every untrusted-input flow.

- **User-typed message** → `POST /api/chat/messages` → zod-validated by `PostMessageRequest` (`packages/channel-web/src/wire/chat.ts:59-77`; max 20 blocks, http-server enforces 1 MiB body cap) → `conversations:append-turn` (validated again via `ContentBlockArraySchema` at store ingress: `packages/conversations/src/store.ts:64-75`) → flows into the LLM (intended sole destination).
- **LLM stream chunks** → bus → SSE → React `MarkdownText` → rendered with `MarkdownTextPrimitive` + `remark-gfm`. No `rehypeRaw`, no raw-HTML injection paths. See `packages/channel-web/src/components/MarkdownText.tsx:11-15`.
- **SSE `reqId` URL param** → matched against `conversations.active_req_id` filtered by `userId` (J9) — foreign reqId returns 404 with no body leak (same response shape as a missing row). Server-side reqId minting at `packages/channel-web/src/server/routes-chat.ts:323`.
- **`chat:run`'s `message` field** stays the convenience flat-string the orchestrator already consumed; no shell / path / HTML interpolation downstream.
- **Stored thinking blocks** — `redacted_thinking` blob is opaque; preserved verbatim through replay so Anthropic compatibility holds (`packages/ipc-protocol/src/content-blocks.ts:43-54`). UI does not render it.
- **JSONB read-back validation** — `rowToTurn` re-validates the JSONB column through `ContentBlockArraySchema` on egress; corrupt rows throw `corrupt-row` rather than escape as malformed types (`packages/conversations/src/store.ts:108-139`).

### Supply chain

- **No new runtime dependencies** in this slice. Conversations reuses `kysely` + `zod` already in the workspace. The runner's chunk emission and IPC client both ride on existing schemas.
- **No new dev dependencies** that pull install-time scripts.
- **Migration safety** — `session_postgres_v2_session_agent.conversation_id` uses `ALTER TABLE … ADD COLUMN IF NOT EXISTS` (`packages/session-postgres/src/migrations.ts:113-114`) — idempotent, forward-only. New `conversations_v1_*` tables use `CREATE TABLE IF NOT EXISTS`.

## Cookie + CSRF posture

Carried unchanged from Week 9.5:

- **`Secure` / `HttpOnly` / `SameSite=Lax`** cookie attributes — set in `packages/http-server/src/cookies.ts:154` (`buildSetCookieHeader`). `Secure` is forced if `SameSite=None`.
- **HMAC-SHA256 signing** with a 32-byte key from `AX_HTTP_COOKIE_KEY`. Tampered values return null without throwing (length / charset short-circuit before `timingSafeEqual`).
- **CSRF middleware** — same-Origin allow-list OR `X-Requested-With: ax-admin` on POST/PUT/PATCH/DELETE.

State-changing chat-flow routes (`POST /api/chat/messages`, `DELETE /api/chat/conversations/:id`) inherit this middleware via `http:register-route` with no chat-flow-specific bypass.

## Known regressions / TODOs

- **Conversation rename UI is a no-op against AX**. `SessionRow.tsx:126-136` and `SessionHeader.tsx:72-85` PATCH `/api/chat/sessions/:id { title }` against an endpoint the AX wire doesn't define. The UI ships green-on-mock and warn-on-real. Tracked as `TODO(week-10-12 follow-up)` in source. Resolution path: next slice ships a rename endpoint OR hides the UI behind a feature flag.
- **Status / diagnostic SSE event types from the prior OpenAI-stub transport are NOT carried forward**. The AX SSE only carries `chunk` and `done` frames (see `packages/channel-web/src/lib/transport.ts`). Reintroduction of richer event types is Week 13+.
- **Multi-replica streaming fan-out**. The in-process `chat:stream-chunk` subscriber + chunk ring buffer is single-replica only. `@ax/eventbus-postgres` swap is Week 13+.
- **Hard-delete admin path for conversations**. Out of scope; soft-delete only (J5). Week 13+.
- **GET /api/chat/agents** returns `agents:list-for-user`'s output as-is, which (per Week 9.5 Task 14 known-issue) does not yet thread `teamIds` for team-agent visibility. Personal agents work; team-agent listing is the same one-line patch already tracked from 9.5.
- **TOCTOU on `clearActiveReqId`**: the clear is compare-and-clear on `(conversation_id, active_req_id)` so a stale callback no-ops; however, a fresh `bind-session` racing with the clear is sequenced by row-level locking inside `appendTurn`. Acceptable for MVP.
- **Team conversation visibility (handoff scenario 4 partial gap).** A team agent's conversation can be CREATED and CONTINUED by the creator, but the conversations store's `:get`/`:list` are scoped by `user_id` only — they do NOT surface a team agent's conversation to OTHER members of the same team. The personal-agent path is fully working. Resolution: a follow-up slice extends `conversations:get`/`:list` to chain `agents:resolve` first; if the agent is a team agent the user can reach, surface the conversation. Tracked in `acceptance-e2e.test.ts` Scenario 4 with a `TODO(team-sharing)` comment.
