# Week 6.5e — `@ax/mcp-client` + `@ax/credentials` Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Dispatch one fresh subagent per task, review between tasks, commit after each.

**Goal:** Add user-configurable MCP servers to AX as a **host-side** plugin so every runner (native, claude-sdk, future pi-session) sees MCP tools through the same `tool:register` / `tool:execute:${name}` surface. Ship `@ax/credentials` alongside it (MCP servers need secrets; credentials plugin also unlocks Week 13+ audit exporters & OAuth flows).

**Architecture:**
- `@ax/mcp-client` runs host-side, reads MCP server configs at boot, opens one MCP **client** per configured server (stdio / streamable-http / sse) via `@modelcontextprotocol/sdk`, discovers tools, fans out to `tool:register` with `executesIn: 'host'`, and registers one `tool:execute:${name}` service hook per remote tool. Tool calls round-trip through MCP wire protocol. Dead servers return structured `MCP_SERVER_UNAVAILABLE` errors; reconnect with backoff.
- `@ax/credentials` registers `credentials:get` / `credentials:set` / `credentials:delete` service hooks, backed by `storage:set`/`storage:get` with AES-256-GCM at-rest encryption. Key sourced from `AX_CREDENTIALS_KEY` env var (32-byte base64 / hex). MCP configs reference credentials by opaque `credential_ref`; inline secrets are rejected.
- Per-agent MCP scoping is **deferred to Week 9.5**. MVP ships a global catalog: all configured MCP servers are visible to all sessions. Clearly flagged temporary in README + code comments.
- Admin HTTP API is **deferred** to Week 9.5 / 10–12 (when `@ax/http-server` lands). MVP provisioning is CLI-driven: `ax-next credentials set <id>` and `ax-next mcp add …` write to storage.

**Tech Stack:** TypeScript (strict), pnpm workspace, vitest, zod, `@modelcontextprotocol/sdk@1.11.5` (exact-pin — rationale: widely deployed, satisfies `@anthropic-ai/claude-agent-sdk@0.2.119`'s peer range of `^1.0.0`, matches the transitive version already in the lockfile from 6.5d; re-pick at implementation time if a security-relevant bump has landed).

**Branch:** `feat/week-6.5e-mcp-client`, branched from tip of 6.5d (which is merged at `33cdcbd` on `main`). Implementation happens in a git worktree (see Task 0).

**Five-invariant audit:**
- **I1 (transport/storage-agnostic hooks):** `tool:register` + `tool:execute:${name}` are unchanged; payloads carry only generic fields. No git/sqlite/k8s vocabulary. ✅
- **I2 (no cross-plugin imports):** `@ax/mcp-client` depends on `@ax/core` only. It discovers `@ax/credentials` through the bus (`credentials:get`). `@ax/credentials` depends on `@ax/core` only and speaks to storage through the bus. ✅
- **I3 (no half-wired plugins):** Both plugins are wired into `@ax/cli`'s `main.ts` in the same PR and lit by the acceptance test. ✅
- **I4 (one source of truth):** MCP tools live in the dispatcher catalog alongside every other tool. MCP configs live in storage under key prefix `mcp-server:`. Credentials live in storage under `credential:`. ✅

**Boundary review output (for PR description):**

- **Payload leaks:** `transport: 'stdio' | 'streamable-http' | 'sse'` is a closed MCP-vocabulary set — not backend-specific. `credential_ref` is an opaque string. ✅

- **Wire surface:** No new IPC actions. Admin API deferred.

**Security review (from `security-checklist` walk):**
- **Sandbox:** `@ax/mcp-client` spawns stdio MCP subprocesses on the host. Argv is the `command` field + `args` array from the validated config (no shell interpolation). Env allowlist is the config's `env` map plus a minimal base (`PATH`, `HOME`) — `process.env[userInput]` nowhere. HTTP/SSE transports default to HTTPS; plain-HTTP URLs log a warning. Maintainers are `modelcontextprotocol` org members (established). Transitive surface walked via `pnpm why` in the plan review. ## Task 2: TDD — `encrypt()` / `decrypt()` primitives

Node's built-in `crypto.createCipheriv` with AES-256-GCM. Random 12-byte IV per value, 16-byte auth tag appended. Output shape: `iv (12) | ciphertext | authTag (16)`, stored as `Uint8Array`. Key is 32 bytes decoded from the `AX_CREDENTIALS_KEY` env var — base64 (44 chars with `=` padding) or hex (64 chars). Callers NEVER pass the key; the module reads it once.

```ts
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';
import { PluginError } from '@ax/core';

const PLUGIN_NAME = '@ax/credentials';
const ALGO = 'aes-256-gcm';
const IV_LEN = 12;
const TAG_LEN = 16;
const KEY_LEN = 32;

export function encryptWithKey(key: Buffer, plaintext: string): Uint8Array {
  if (key.length !== KEY_LEN) {
    throw new PluginError({
      code: 'invalid-key',
      plugin: PLUGIN_NAME,
      message: `encryption key must be ${KEY_LEN} bytes`,
    });
  }
  const iv = randomBytes(IV_LEN);
  const cipher = createCipheriv(ALGO, key, iv);
  const ct = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  const tag = cipher.getAuthTag();
  return new Uint8Array(Buffer.concat([iv, ct, tag]));
}

export function decryptWithKey(key: Buffer, blob: Uint8Array): string {
  if (key.length !== KEY_LEN) {
    throw new PluginError({
      code: 'invalid-key',
      plugin: PLUGIN_NAME,
      message: `decryption key must be ${KEY_LEN} bytes`,
    });
  }
  if (blob.length < IV_LEN + TAG_LEN) {
    throw new PluginError({
      code: 'invalid-ciphertext',
      plugin: PLUGIN_NAME,
      message: 'ciphertext too short',
    });
  }
  const buf = Buffer.from(blob);
  const iv = buf.subarray(0, IV_LEN);
  const tag = buf.subarray(buf.length - TAG_LEN);
  const ct = buf.subarray(IV_LEN, buf.length - TAG_LEN);
  const decipher = createDecipheriv(ALGO, key, iv);
  decipher.setAuthTag(tag);
  // Error intentionally opaque — never mentions plaintext or input.
  try {
    return Buffer.concat([decipher.update(ct), decipher.final()]).toString('utf8');
  } catch (_err) {
    throw new PluginError({
      code: 'decrypt-failed',
      plugin: PLUGIN_NAME,
      message: 'authentication tag mismatch',
    });
  }
}

export function parseKeyFromEnv(raw: string): Buffer {
  // Try hex first (64 lowercase/upper-case hex chars), then base64.
  if (/^[0-9a-fA-F]{64}$/.test(raw)) {
    return Buffer.from(raw, 'hex');
  }
  const b64 = Buffer.from(raw, 'base64');
  if (b64.length === KEY_LEN) return b64;
  // Never echo the invalid input back — it may be a partial/mistyped real key.
  throw new PluginError({
    code: 'invalid-key',
    plugin: PLUGIN_NAME,
    message: `AX_CREDENTIALS_KEY must be a 32-byte key (64 hex chars or 44 base64 chars)`,
  });
}
```

**Step 3: Implement** (`packages/credentials/src/plugin.ts`):

```ts
import { PluginError, type Plugin } from '@ax/core';
import { encryptWithKey, decryptWithKey, parseKeyFromEnv } from './crypto.js';

const PLUGIN_NAME = '@ax/credentials';
const CREDENTIAL_KEY_PREFIX = 'credential:';
const ID_RE = /^[a-z0-9][a-z0-9_.-]{0,127}$/;

function storageKey(id: string): string {
  return `${CREDENTIAL_KEY_PREFIX}${id}`;
}

function validateId(id: unknown): string {
  if (typeof id !== 'string' || !ID_RE.test(id)) {
    throw new PluginError({
      code: 'invalid-payload',
      plugin: PLUGIN_NAME,
      message: `credential id must match ${ID_RE.source}`,
    });
  }
  return id;
}

export function createCredentialsPlugin(): Plugin {
  return {
    manifest: {
      name: PLUGIN_NAME,
      version: '0.0.0',
      registers: ['credentials:get', 'credentials:set', 'credentials:delete'],
      calls: ['storage:get', 'storage:set'],
      subscribes: [],
    },
    async init({ bus }) {
      const raw = process.env.AX_CREDENTIALS_KEY;
      if (raw === undefined || raw === '') {
        throw new PluginError({
          code: 'missing-env',
          plugin: PLUGIN_NAME,
          message:
            'AX_CREDENTIALS_KEY is required (32 bytes, 64 hex chars or 44 base64 chars)',
        });
      }
      const key = parseKeyFromEnv(raw);

      bus.registerService<{ id: string; value: string }, void>(
        'credentials:set',
        PLUGIN_NAME,
        async (ctx, input) => {
          const id = validateId(input.id);
          if (typeof input.value !== 'string') {
            throw new PluginError({
              code: 'invalid-payload',
              plugin: PLUGIN_NAME,
              message: `credential value must be a string`,
            });
          }
          const blob = encryptWithKey(key, input.value);
          await bus.call('storage:set', ctx, { key: storageKey(id), value: blob });
        },
      );

      bus.registerService<{ id: string }, { value: string }>(
        'credentials:get',
        PLUGIN_NAME,
        async (ctx, input) => {
          const id = validateId(input.id);
          const got = await bus.call<{ key: string }, { value: Uint8Array | undefined }>(
            'storage:get',
            ctx,
            { key: storageKey(id) },
          );
          if (got.value === undefined) {
            throw new PluginError({
              code: 'credential-not-found',
              plugin: PLUGIN_NAME,
              message: `no credential with id '${id}'`,
            });
          }
          // decryptWithKey never echoes plaintext on failure — see crypto.ts.
          const value = decryptWithKey(key, got.value);
          return { value };
        },
      );

      bus.registerService<{ id: string }, void>(
        'credentials:delete',
        PLUGIN_NAME,
        async (ctx, input) => {
          const id = validateId(input.id);
          // Tombstone via empty value; `storage:get` returns `undefined` for
          // missing keys, so write a zero-length Uint8Array then verify via
          // our own lookup. Simpler: reuse storage:set with empty buffer, and
          // have `credentials:get` treat a post-decrypt-failed zero-length as
          // deleted. Cleaner still: add `storage:delete` in a future slice.
          // For MVP: overwrite with a tombstone marker.
          const tombstone = encryptWithKey(key, '');
          // Mark tombstone with a version byte so `credentials:get` can
          // distinguish "empty string" from "deleted". v0 prefix byte = 0x00.
          // (Skipped for MVP — we treat empty plaintext as deleted. See
          // README "Known limits".)
          await bus.call('storage:set', ctx, { key: storageKey(id), value: tombstone });
        },
      );
    },
  };
}
```

> **Note to implementing subagent:** the `credentials:delete` MVP uses a tombstone (encrypted empty string) because `@ax/storage-sqlite` doesn't expose `storage:delete` yet. Update `credentials:get` to throw `credential-not-found` when the decrypted value is empty — match the test. If the empty-string semantics feel wrong, fall back to a sentinel byte prefix; either works as long as the tests in Task 3 pass verbatim.

# Appendix C — Why two packages in one slice

MCP needs a credentials store; credentials store needs a caller. Shipping them separately creates a half-wired state on one side or the other for however many days the second PR sits behind the first. That violates the Half-Wired Code Policy in CLAUDE.md — hence one slice, two packages, one PR.

If the PR grows unwieldy during review, split at the `Phase A → Phase B` boundary in this plan: Phase A is independently reviewable (it's just a keyed, encrypted KV store) and the Phase B PR can stack on top.
