# Week 6.5e handoff — `@ax/mcp-client` (host-side MCP hosting)

**For:** session starting after Week 6.5d (claude-sdk runner) lands.
**Previous slices:** Weeks 1–2, 3, 4–6, 6.5a (topology shift), 6.5d (claude-sdk runner).
**Successors:** Week 7–9 (k8s deployment) — MCP connections need to survive replica restarts; MVP admin UI (Week 9.5) — user-configurable MCP server list per agent.

Skipped from the original 6.5 design: 6.5b (openrouter + streaming) and 6.5c (pi-session runner). See `project_mvp_direction.md` in memory for rationale.

---

## Goal

Add user-configurable MCP servers to AX in a way that's runner-agnostic: every runner (native, claude-sdk, future pi-session) sees MCP tools through the same `tool:register` / `tool:execute:${name}` surface as every other tool.

Why host-side, not runner-side:

- **Runner-agnostic by construction.** Claude-sdk has native `mcpServers` support; pi-coding-agent may or may not; our native runner doesn't. Hosting MCP clients on the host means one implementation covers all three and any future runner.
- **One source of truth (invariant 4).** The tool catalog lives in `@ax/tool-dispatcher`. If MCP tools live in runner-land, the catalog forks per runner.
- **Uniform subscriber semantics (invariant 1).** `tool:pre-call` / `tool:post-call` fire for MCP tools the same as bash. Canary, audit, memory see everything with no special-casing.
- **Capability minimization (invariant 5).** MCP servers run in host-land where we already have credential handling; sandbox doesn't need network egress to MCP endpoints.

---

## Deliverables

- **`@ax/mcp-client`** — host-side plugin. At init, reads MCP server configs from storage. For each enabled server, opens a client connection (stdio/http/sse per MCP spec). Calls `tool:register` for each remote tool the server advertises, with `executesIn: 'host'`. Registers `tool:execute:${toolName}` service hook per tool, whose handler marshals the call over MCP and returns the result.
- **`@ax/credentials`** — pulled forward from Week 13+. MCP servers need API keys / OAuth tokens. One credentials plugin unlocks MCP + later audit exporters + OAuth flows. Minimal shape: `credentials:get(key) → value` service hook backed by `storage:set/get` with at-rest encryption.
- **`@ax/mcp-admin-api`** *(or folded into a larger admin plugin later)* — registers HTTP routes on `@ax/http-server` for CRUD on MCP server configs. Requires Week 10–12's `@ax/http-server` to exist, OR ships a minimal standalone HTTP listener for pre-channel use. Decide below.

## Scope decisions to make while writing the plan

1. **MCP client library.** Use `@modelcontextprotocol/sdk` (official TypeScript SDK) rather than hand-rolling. It handles stdio/http/sse transports and the JSON-RPC layer.

2. **Transport set at MVP.** Support which?
   - **(a)** Stdio only (simplest, covers most community MCP servers).
   - **(b)** Stdio + HTTP + SSE.
   - **Recommendation:** (b). HTTP/SSE lets users bring remote MCP services without needing them co-located. The SDK gives us all three essentially free.

3. **MCP server lifecycle.** Stdio servers are child processes of the host. When?
   - **(a)** At plugin init (eager) — all configured servers spawn at host boot.
   - **(b)** Lazy — spawn on first tool call, keep alive with an idle timer.
   - **(c)** Per-session — spawn when a chat that uses the server starts, tear down on `chat:end`.
   - **Recommendation:** (a) for MVP. Eager simplifies the tool catalog (known at boot), avoids cold-start latency on first call, and matches how the architecture already handles the tool registration phase. Revisit if a heavy/slow MCP server makes boot painful.

4. **Per-agent MCP scoping.** Feature 2 says "users configure MCP connections in admin interface." Feature 6 says "personal vs team agents." So MCP configs belong to agents, not to the global host.
   - Implications: the tool catalog is per-agent-session, not global. `tool:list` fired during `sandbox:open-session` needs to filter to the calling agent's MCP allowlist.
   - This changes `@ax/tool-dispatcher`'s current "one global catalog" assumption — the dispatcher needs to maintain per-agent filtered views.
   - **Recommendation:** punt agent-scoping to Week 9.5 (when `@ax/agents` lands). For MVP `@ax/mcp-client` ships with global scope: all MCP servers visible to all sessions. Clearly flagged as temporary. Week 9.5 adds the per-agent filter.

5. **Credential storage.** MCP server configs include secrets (API keys for HTTP/SSE servers, sometimes env vars for stdio servers).
   - Store encrypted at rest via `@ax/credentials`. Reference by credential ID from the MCP server config, not inline value.
   - Encryption key source: env var (`AX_CREDENTIALS_KEY`) for MVP. KMS integration is a Week 13+ candidate.

6. **Dead MCP server handling.** Stdio server crashes, HTTP endpoint goes down.
   - Mark server unhealthy; its tools become temporarily unavailable.
   - `tool:execute:${name}` returns a structured error (`MCP_SERVER_UNAVAILABLE`) that the runner surfaces to the model as a tool error, not a chat-terminating failure.
   - Reconnect with backoff.

7. **Admin API surface.** Minimal CRUD:
   - `POST /admin/mcp-servers` — create config (name, transport, command/url, env, credential refs).
   - `GET /admin/mcp-servers` — list.
   - `PATCH /admin/mcp-servers/:id` — update, enable/disable.
   - `DELETE /admin/mcp-servers/:id`.
   - `POST /admin/mcp-servers/:id/test` — attempt connection, return status + discovered tools.
   - Auth: this surface requires admin identity. Since `@ax/auth` is Week 9.5, MVP can either (a) punt auth on the admin API to Week 9.5 and use a bootstrap bearer token, or (b) ship a minimal bearer-token config read from env. **Recommendation:** (b) — one env-var admin token for pre-9.5, replaced by real auth in 9.5.

## Boundary review

- **`tool:register` payload:** already specs `{ name, description, inputSchema, executesIn }`. MCP tools use this unchanged. ✅
- **`tool:execute:${name}` service hook:** generic service hook per tool name, already part of the dispatcher design. MCP tools register via `tool:register` then implement the corresponding service hook. ✅
- **New payload fields in admin API:** `transport: 'stdio' | 'http' | 'sse'` — closed set, no backend-specific leak. `credential_ref` — opaque reference, no credential storage vocabulary leaks. ✅
- **Alternative impl this hook surface could have:** any other dynamic-tool-source plugin (e.g., a future OpenAPI-to-tool plugin, a plugin that registers a fixed set of LLM-as-tool calls). The `tool:register` + `tool:execute:${name}` pair isn't MCP-specific, which is the point. ✅

## Security — `security-checklist` required (heavy)

Three threat models all apply:

- **Sandbox escape:** N/A directly — MCP runs host-side, not in the sandbox. But an MCP server returning malicious tool output is prompt injection into the model running in the sandbox.
- **Prompt injection:** MCP server output is **third-party model-fed input**. Treat as untrusted at every hop. `tool:post-call` subscribers (canary) must scan MCP output the same as bash output. Document in `@ax/mcp-client`'s SECURITY.md that MCP servers can be hostile.
- **Supply chain:** `@modelcontextprotocol/sdk` is a new dependency. Community MCP servers are arbitrary npm packages or remote services — users bring their own. The plugin never auto-installs MCP servers; configs reference already-available commands or URLs.

Additional concerns:
- **Credential exposure.** `@ax/credentials` must never log decrypted values. Redact in error paths.
- **Stdio MCP process isolation.** Stdio MCP servers run as host subprocesses. Consider ulimits / resource caps, especially for community servers.
- **HTTP MCP TLS.** HTTP transport must default to HTTPS; log a warning for plain-HTTP configs. No cert pinning at MVP but note it as Week 13+.
- **Input schema handling.** MCP servers declare their own JSON Schema for tool inputs. We pass this through to the model via `tool:list`. Malicious schemas could confuse the model or embed prompt-injection hints in descriptions. Render tool descriptions as untrusted content, not trusted system text.

## Acceptance test

Automated:
1. Start host with one stdio MCP server config (e.g., a minimal test MCP server included in `@ax/test-harness`).
2. Verify `@ax/mcp-client` spawned the subprocess and called `tool:register` for each advertised tool.
3. Run a chat that causes the model to call one of the MCP tools.
4. Assert `tool:pre-call` fired, `tool:execute:${name}` dispatched to the MCP client, call round-tripped through MCP wire protocol, response returned.
5. Assert `tool:post-call` fired with the result visible.
6. Kill the MCP subprocess mid-test; verify next call returns `MCP_SERVER_UNAVAILABLE` error without terminating the chat.

Manual: configure a real community MCP server (e.g., `mcp-server-filesystem` in read-only mode), run a chat that uses it, verify tools appear in the catalog and execute correctly.
