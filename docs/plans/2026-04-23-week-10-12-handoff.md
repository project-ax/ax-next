## Goal

```
Week 10-12 (MVP) — Web chat + conversation persistence
  • @ax/channel-chat-ui (port assistant-ui adapter)
  • @ax/conversations (persisted multi-turn history store)
  • Streaming (chat:stream-chunk plumbing) if not already landed in 6.5d

```

Translation: ship the browser chat surface on top of 9.5's tenant primitives, with durable conversation history so reloads don't lose context. Slack, audit, and canary scanner are **deferred to Week 13+** — scope narrowed to the shortest path to "user can actually use this from a browser." Memory stays deferred to Week 13+ via Strata.

## Deliverables

- `@ax/channel-web` (extends 9.5's scaffold; not a new package). The assistant-ui adapter is already ported and `LoginPage` is wired to `@ax/auth-oidc`. Remaining work: SSE streaming via filtered `chat:stream-chunk` subscriber; chat-flow HTTP routes registered against `@ax/http-server`; agent picker (using the existing `AgentMenu`) reading the user's agents via `@ax/agents`; conversation history loaded from `@ax/conversations` on mount; new user messages posted and assistant turns appended as they complete. **Admin operations** continue to live in `packages/channel-web/src/components/admin/` as user-menu entries (agent CRUD, MCP server config, team management if teams shipped in 9.5), calling 9.5's `/admin/*` endpoints — no separate admin app. Authz is enforced at the API layer, not by hiding menu items.
- `@ax/conversations` — persisted conversation store. Postgres-backed (depends on `@ax/storage-postgres` from Week 7–9). Registers `conversations:create`, `conversations:append-turn`, `conversations:get`, `conversations:list` service hooks. Each conversation is tenant-scoped by `user_id` + `agent_id`; ACL check on every read/write. Subscribes to `chat:turn-end` to append completed turns without the channel plugin having to do it explicitly.
- **Streaming plumbing, if not landed in 6.5d.** `event.stream-chunk` IPC event + `chat:stream-chunk` subscriber hook + `@ax/channel-chat-ui` SSE consumer. Claude-sdk emits stream chunks natively via `SDKMessage` iteration — the work is the IPC + subscriber plumbing, not the runner side.

## Scope decisions to make while writing the plan

1. **`chat:stream-chunk` hook.** If already landed as a 6.5d plumb, skip this decision. Otherwise: fires from `@ax/ipc-server` on receipt of `event.stream-chunk` from the runner; `@ax/channel-chat-ui` subscribes filtered by `reqId`. Sandbox-side, claude-sdk's `SDKMessage` iteration is already producing the chunks.

2. **Chat UI agent picker.** When a user opens the chat UI, they need to pick which agent to chat with. Options: dropdown of the user's accessible agents; agent-per-URL (`/chat/<agent-slug>`); session-persisted last agent. Any is fine; recommendation is agent-per-URL so sharing a link is unambiguous.

3. **Conversation ↔ session relationship.** A conversation can outlive the underlying sandbox session (reload a browser → same conversation, new or resumed sandbox). Options:
   - **(a)** One sandbox session per conversation, resumed on reload. Requires resumable sandboxes — 6.5a's pull-based inbox + durable session store supports this, but runner needs to replay history from `@ax/conversations` on startup.
   - **(b)** New sandbox session per browser page-load, conversation is just the message log. Simpler but loses in-flight context.
   - **Recommendation:** (a). Reload-during-a-tool-call should not interrupt the tool. The runner fetches conversation history via `conversations:get` at session start.

4. **Conversation message shape.** Conversations store tuples of `(role, content_blocks)` per turn. Tool-use + tool-result blocks are part of the content (Anthropic tool-result block shape or similar — pick one that round-trips cleanly through the SDK). Assistant thinking blocks: store? show in UI? Recommendation: store (for faithful replay into the SDK on resume) but hide by default in UI.

5. **Conversation deletion / archival.** Users will want to delete conversations. Soft-delete with a `deleted_at` column + filter in `conversations:list`. Hard-delete as a Week 13+ admin cleanup task. Recommendation: ship soft-delete in this slice; wire a hard-delete admin API later.

## Deferred from this slice — Week 13+

Explicit deferrals (flag in the plan so they don't get "just a little bit here" scope creep):

- **`@ax/channel-slack`** — not in MVP. When it lands: Slack inbound via socket mode (one replica holds the socket per workspace); Slack outbound via HTTPS to `slack.com/api` from any replica. No multi-replica `channel:outbound` eventbus coordination needed — outbound is stateless HTTP. Slack user→AX user mapping + workspace/channel→agent mapping are new primitives.
- **`@ax/audit`** — not in MVP. When it lands: subscribes to `chat:end`, `workspace:applied`, `tool:post-call`, `auth:user-signed-in`. Audit rows carry `user_id` + `agent_id`. Decision deferred: `storage:set` vs dedicated `audit:write` service hook.
- **`@ax/scanner-canary`** — not in MVP. When it lands: subscribes to `workspace:pre-apply` (secret veto) and `llm:pre-call` (redaction before content reaches the model, per D4). Tool output scanning at `llm:pre-call` (aggregate view), not per-tool-call. Port regex set from legacy.

**Risk of shipping MVP without canary/audit:** a personal-use-or-small-team deployment has no secret-leak veto and no forensic trail. Acceptable for an internal MVP among trusted users; **do not** open to wider access without at least `@ax/scanner-canary` in place.

## Security — `security-checklist` required

- **`@ax/channel-web`** — untrusted browser input, CSRF, SSE connection-management, auth handshake against 9.5's `@ax/auth-oidc` session cookies.
- **`@ax/conversations`** — tenant isolation is the whole concern. Every `conversations:*` service hook takes a `user_id` and validates agent access via `agents:resolve` before reading or writing. No cross-tenant leak through `conversations:list` (must filter). Conversations can contain sensitive content — redaction is a Week 13+ canary concern; storage-at-rest encryption inherits postgres defaults.

## Legacy helpers to port (read-only `~/dev/ai/ax/`)

- assistant-ui adapter shell — **already ported into `@ax/channel-web` during 9.5**; nothing to re-port.
- Conversation rendering / streaming hooks specific to multi-turn history (chunk reassembly, tool-result block rendering) — port as needed when wiring `@ax/conversations`.
- **Skip:** Slack channel, canary regex, cortex memory (all deferred). 9.5 already handles session tokens / auth middleware.

## Acceptance test for Week 10–12

End-to-end: k8s deployment, User A signs in, picks their personal agent, sends a chat message from the browser, gets a streamed response in real time. User A reloads the page mid-conversation; conversation history reappears and a new message continues the same thread. User A opens a second tab with the same agent URL; both tabs see the same conversation. User A shares a team agent's conversation URL with User B (member of the same team); User B loads it and can continue. User C (not in the team) loading the same URL → 403.

This is the MVP-launch milestone. Users can actually use the system from a browser with durable conversation history.
