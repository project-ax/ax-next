# Week 6.5 — Agent sandboxing + multi-provider LLM design

**Status:** design, pre-implementation-plan.
**Predecessors:** Weeks 1–2, 3, 4–6.
**Successors:** Week 7–9 (k8s deployment shape) consumes this contract.

---

## Goal

Make the agent loop pluggable and move it out of the host process, so that:

1. Users can pick their LLM provider (Anthropic, OpenRouter, future OpenAI/Gemini) for cost savings.
2. Users can pick their agent framework (our own loop, `@mariozechner/pi-coding-agent` / pi-session, Anthropic's `@anthropic-ai/claude-agent-sdk`) for capability tradeoffs.
3. Agent execution runs in a sandbox pod in production and a subprocess-sandbox in local dev, with **one code path** — the same IPC protocol in every mode.
4. Either host or sandbox can terminate unexpectedly without losing the session (any host replica can serve any sandbox; durable inbox; clean reconnect).

## Motivation

Current Week 4-6 shipped a host-local chat loop (`packages/core/src/chat-loop.ts`). That's the wrong topology for production — tools execute in the host process, which violates the capability-minimization invariant (I5). It also doesn't compose with pi-session or claude-agent-sdk: both are SDK-owned-loop frameworks that run as their own process with tools resolved via callbacks, not providers pluggable into our loop.

v1 has already proven the target shape (host↔sandbox HTTP IPC, pull-based inbox, per-session workspace commits via git). We're porting the pattern and fixing v1's two known weaknesses: in-memory inbox bound to a single host replica (no failover) and no graceful cancellation path.

## Scope: four sequenced slices

Each slice is independently reviewable with a clean "nothing observable changes except the intended thing" acceptance. Smaller than 6.5a-as-one-blob (which would be bigger than Week 4–6 was) and targets risk where it actually lives — the topology shift dominates difficulty; providers and alternative runners are additive.

Still `@ax/llm-anthropic`; no streaming. | Existing Week 4–6 chat works end-to-end with the agent loop running in a subprocess sandbox; all existing subscribers fire correctly across the IPC boundary; no observable behavior change vs. Week 4–6. |

## Core design decisions

### D1. Agent loop runs in the sandbox, not the host

The chat loop that lives in `packages/core/src/chat-loop.ts` today moves into a sandbox-side runner binary. The host orchestrates but does not execute the loop.

Why:
- Tool execution inherits the sandbox's isolation boundary (I5).
- pi-session and claude-agent-sdk both own their internal loops. A pluggable loop on the host can't embed them cleanly.
- v1 runs this way; the pattern is validated.

Cost: we lose in-process speed (every LLM call is an IPC round-trip). Acceptable — LLM latency dominates; IPC overhead is sub-millisecond over a unix socket.

### D2. Sandbox is always the HTTP client; host never pushes

All IPC traffic is sandbox-initiated. The host runs one HTTP listener (unix socket locally, cluster HTTP in k8s); the sandbox has no inbound port and serves nothing.

Why:
- Single auth boundary (every inbound request to host is token-verified; sandbox grants no new capabilities).
- Sandbox is the untrusted side; fewer listening sockets = smaller attack surface.
- Matches v1's pattern.

### D3. Host→sandbox delivery is pull-based via durable inbox

When the host needs to deliver something to a running sandbox (follow-up user message, cancellation), it writes to a durable inbox keyed by `session_id`. The sandbox long-polls.

| Preset | Inbox backing |
|---|---|
| Local / single-host | In-memory queue in `@ax/session-inmemory` |
| k8s (Week 7–9) | Postgres table in `@ax/session-postgres`, with `LISTEN/NOTIFY` wakeup on long-polls |

Consequences:
- Any host replica can serve any sandbox's poll. No sticky sessions, no replica affinity.
- Host replica can crash mid-poll — sandbox's next poll hits a different replica, continues.
- Sandbox pod eviction detected by absence of polls beyond a heartbeat threshold; session marked terminated.
- No NATS, no JetStream, no second message broker. Inbox is a table with long-poll.

Cancellation is an inbox entry of type `cancel`. Runner handles it gracefully and exits.

### D4. `tool:post-call` is observation-only; `llm:pre-call` is the transformation point

`claude-agent-sdk`'s `PostToolUse` hook is observation-only in practice — the `updatedMCPToolOutput` field is declared in the type but not read by `cli.js`. To keep subscriber behavior uniform across all three runners, we adopt observation-only for `tool:post-call` everywhere.

Subscribers that need to **modify** content the model will see (canary redaction, memory recall injection, prompt restructuring) shift to `llm:pre-call`, which fires host-side right before every LLM request.

Subscribers that just **observe** tool calls (audit, telemetry, memory extraction) stay on `tool:post-call`.

Why this is actually better than tool-level redaction:
- `llm:pre-call` transformation catches secrets from every source, not just fresh tool output.
- Audit logs retain the raw, un-redacted tool output for forensics.
- Uniform across runners, tools, and execution locations (sandbox-local or host-mediated).

### D5. Three runners, one IPC contract

| Runner | Slice | Tool path | LLM path |
|---|---|---|---|
| `@ax/agent-native-runner` | 6.5a | Direct handlers with IPC pre/post | `llm.call` IPC directly |
| `@ax/agent-pi-session-runner` | 6.5c | pi-session `customTools` callbacks with IPC pre/post | `registerApiProvider` (single `ax-ipc` registration) routes through `llm.call` IPC |
| `@ax/agent-claude-sdk-runner` | 6.5d | Hybrid: claude-sdk built-ins + in-process MCP handlers, both wired via `canUseTool` + `hooks.PreToolUse`/`PostToolUse` | `ANTHROPIC_BASE_URL` points at host-side Anthropic-format proxy → `llm.call` IPC |

All three fire the same host-side subscriber hooks via the same IPC events. Host observers (audit, canary, memory) don't know which runner is active.

## The sandbox ladder

Three sandbox implementations, identical `sandbox:open-session` service hook contract.

| Plugin | Isolation | When | 6.5a? |
|---|---|---|---|
| `@ax/sandbox-subprocess` | None — subprocess of host, shared FS & network | CI, fastest dev iteration, integration tests | ✅ extended |
| `@ax/sandbox-docker` | Docker container / Apple container — real FS + network isolation | Realistic local dev | Stretch |
| `@ax/sandbox-k8s` | k8s pod — full isolation, RBAC, NetworkPolicy | Production | Week 7–9 |

All three spawn the runner binary, inject env (`AX_IPC_SOCKET` or `AX_IPC_URL`, `AX_SESSION_ID`, `AX_AUTH_TOKEN`), return a handle with `kill()`. The runner binary is identical in every mode. `@ax/sandbox-subprocess`'s total lack of isolation is fine for dev and CI; production never loads it.

## IPC protocol

### Transport

HTTP+JSON. Transport varies by sandbox plugin:

| Sandbox plugin | Transport |
|---|---|
| `@ax/sandbox-subprocess` | Unix socket at `${AX_IPC_SOCKET}` |
| `@ax/sandbox-docker` | Unix socket bind-mounted into container |
| `@ax/sandbox-k8s` | Cluster HTTP — host exposes a Service reachable from pods |

Same wire format, same actions. The runner reads `AX_IPC_SOCKET` (socket path) or `AX_IPC_URL` (http URL), otherwise indistinguishable.

### Auth

Per-session bearer token issued at `sandbox:open-session` time. Stored server-side in the session store (postgres in k8s, in-memory locally). Any host replica resolves the token against the store.

Token lifetime = sandbox session lifetime. No per-turn rotation (v1's per-turn rotation was a workaround for its in-memory-replica problem, which we fix structurally).

Revocation: mark session terminated in the store; subsequent IPC calls with that token return 401.

### Sandbox → host actions (sync RPC)

| Action | Payload | Response | Host-side effect |
|---|---|---|---|
| `llm.call` | `{ messages, tools?, model?, maxTokens?, temperature? }` | `{ assistantMessage, toolCalls[], stopReason, usage? }` | Fires `llm:pre-call` (transform), calls `llm:call` service hook, fires `llm:post-call` (observe) |
| `tool.pre-call` | `{ call: { id, name, input } }` | `{ verdict: 'allow' \| 'reject', modifiedCall?, reason? }` | Fires `tool:pre-call` subscriber chain |
| `tool.execute-host` | `{ call: { id, name, input } }` | `{ output }` | Dispatches `tool:execute:${name}` service hook (used for `executesIn: 'host'` tools) |
| `tool.list` | `{}` | `{ tools: ToolDescriptor[] }` | Returns the aggregated catalog |
| `workspace.commit-notify` | `{ parentVersion?, commitRef, message }` | `{ accepted: true, version, delta } \| { accepted: false, reason }` | Host's `@ax/workspace-git` fetches delta, fires `workspace:pre-apply` (veto-capable) → on accept: promotes + fires `workspace:applied` |
| `session.next-message` | `?cursor=N` query param; long-poll up to 30s | `{ type: 'user-message' \| 'cancel' \| 'timeout', payload?, cursor }` | Reads inbox, long-poll wakeup on NOTIFY |

### Sandbox → host events (fire-and-forget)

| Action | Payload | Host-side effect |
|---|---|---|
| `event.stream-chunk` | `{ reqId, text, kind: 'text' \| 'thinking' }` | Fires `chat:stream-chunk` subscriber (chat-ui SSE, CLI tty) |
| `event.tool-post-call` | `{ call, output, durationMs }` | Fires `tool:post-call` subscriber chain (observation only, per D4) |
| `event.turn-end` | `{ reqId, reason: 'user-message-wait' \| 'error' \| 'complete', usage? }` | Fires `chat:turn-end` subscriber |
| `event.chat-end` | `{ outcome: ChatOutcome }` | Fires `chat:end` subscriber |

### Host → sandbox actions

**None.** Host never pushes. All host→sandbox communication is via `session.next-message` responses.

### Error model

Every RPC response that isn't a success carries `{ error: { code, message } }`. Codes are enum-ish:
- `SESSION_INVALID` — token rejected, sandbox should exit
- `HOST_UNAVAILABLE` — retryable (replica died; sandbox reconnects)
- `VALIDATION` — payload schema error
- `HOOK_REJECTED` — subscriber chain rejected (e.g., `tool:pre-call` verdict reject)
- `INTERNAL` — unexpected host-side error

Connection-level errors (socket close, HTTP error): sandbox retries with exponential backoff, capped at ~30s. Host session persists in the store; the next successful request resumes.

### Schema location

Every action schema lives in `@ax/ipc-protocol` — one pnpm package, pure types + Zod, no runtime. Both host-side plugins and sandbox-side runner import it. Single source of truth for the wire.

## Runner comparison

### `@ax/agent-native-runner` (6.5a; streaming added in 6.5b)

Simplest. A faithful port of current `chat-loop.ts` logic into the sandbox.

```
while (!done) {
  for (let msg = await ipcCall('session.next-message', {cursor}); msg.type !== 'cancel';) {
    history.push({ role: 'user', content: msg.payload })
    loop:
    while (true) {
      const response = await ipcCall('llm.call', { messages: history, tools: catalog })
      history.push(response.assistantMessage)
      fireStreamChunk(response)  // if streaming
      if (response.toolCalls.length === 0) break loop
      for (const call of response.toolCalls) {
        const pre = await ipcCall('tool.pre-call', { call })
        if (pre.verdict === 'reject') { history.push(toolErrorMsg(pre.reason)); continue }
        const output = await executeToolLocally(pre.modifiedCall ?? call) ?? await ipcCall('tool.execute-host', { call })
        ipcEvent('event.tool-post-call', { call, output })
        history.push(toolResultMsg(call, output))
      }
    }
    ipcEvent('event.turn-end', { reason: 'user-message-wait' })
    cursor = msg.cursor
  }
}
```

Non-streaming for 6.5a. Streaming lands in 6.5b alongside `chat:stream-chunk` subscriber work.

### `@ax/agent-pi-session-runner` (6.5c)

Wraps `createAgentSession()` from `@mariozechner/pi-coding-agent`. Works unchanged with every `@ax/llm-*` plugin the host supports — the runner is provider-agnostic.

**LLM transport — single custom api, honest model name:**

pi-ai's `registerApiProvider` is one-api-per-call, but pi-ai itself does not pre-format requests into Anthropic/OpenAI wire shape — the registered transport receives pi-ai's normalized `Context` (messages, tools, systemPrompt) plus the `Model` object. So we register ONCE with a custom api name and handle every backing provider uniformly:

```ts
// Runner init (sandbox-side)
registerApiProvider({
  api: 'ax-ipc',
  stream: async (ctx, model, options) => {
    const response = await ipcCall('llm.call', {
      messages: normalizeMessages(ctx.messages),
      tools: ctx.tools,
      model: model.name,           // honest: configured model identifier passes through
      maxTokens: options.maxTokens,
      temperature: options.temperature,
    });
    return toPiAiResponse(response);
  },
  streamSimple: /* same */,
});

// Build the Model pi-session will see
const model = {
  api: 'ax-ipc',                   // routes to our transport
  provider: 'ax',
  name: process.env.AX_MODEL,      // e.g. 'openrouter/gpt-4.1' or 'anthropic/claude-opus-4-7'
  baseUrl: '',                     // unused by our transport
};
```

Host's `llm:call` service hook reads the `model` field from the IPC payload and routes to the configured `@ax/llm-*` plugin. OpenRouter, Anthropic, or any future provider — runner never changes.

**Do not hardcode Model.name.** v1 effectively hardcoded a Claude model name regardless of the backing call; that creates a telemetry/logging mismatch. The configured identifier must flow through honestly.

**Tools:** convert our `tool:list` catalog into pi-session's `customTools: ToolDefinition[]`. Each definition's `execute` callback does the IPC pre/post dance (per D4).

**Event translation:** subscribe to `session.subscribe()`; map pi-session events to our IPC events (`text_delta` → `event.stream-chunk`, `tool_call` is handled by our callback, `message_end` → `event.turn-end`).

**Message buffering:** pi-session uses `session.prompt(text)` once per turn, then `waitForIdle()`. Our runner loop pulls next-message from the inbox and calls `prompt`.

### `@ax/agent-claude-sdk-runner` (6.5d)

Wraps `@anthropic-ai/claude-agent-sdk`'s `query()`.

**Built-in tool policy:** keep `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `TodoWrite`. Disable `WebFetch`, `WebSearch`, `Skill`, `Task`. The kept ones run inside the sandbox process tree (claude-sdk's `sdk.mjs` → `cli.js` → tool subprocesses are all descendants of the runner, so isolation holds).

**Tool wiring:**
- `canUseTool` callback → IPC `tool.pre-call`. Works for both built-ins and MCP tools. Deny → `{behavior: 'deny', message}`; rewrite → `{behavior: 'allow', updatedInput}`.
- `hooks.PostToolUse` → IPC `event.tool-post-call` (fire-and-forget, observation-only per D4).
- `mcpServers` includes an in-process `createSdkMcpServer` registering our `executesIn: 'host'` tools (memory, web-via-host, etc., as they arrive). Each MCP handler does the IPC pre/post and returns output.

**LLM transport:** `env: { ANTHROPIC_BASE_URL: '${AX_LLM_PROXY_URL}', ANTHROPIC_API_KEY: 'ax-proxy' }`. The host runs a small Anthropic-format proxy endpoint that translates `POST /v1/messages` into the `llm.call` IPC action. This is a new host-side component introduced in 6.5b, not 6.5a.

**Event translation:** iterate the `AsyncGenerator<SDKMessage>`. `SDKAssistantMessage` with text content → `event.stream-chunk`. `SDKResultMessage` → `event.turn-end` + `event.chat-end` on final result.

## Tool registry

### `ToolDescriptor` shape

```ts
type ToolDescriptor = {
  name: string;              // e.g., 'bash', 'memory.recall'
  description: string;
  inputSchema: JsonSchema;   // JSON Schema Draft 2020-12
  executesIn: 'sandbox' | 'host';
}
```

### Registration flow

1. Plugin declares `calls: ['tool:register']` in its manifest.
2. In its `init()`, plugin calls `bus.call('tool:register', { ...descriptor })` for each tool it contributes.
3. Boot-time cycle detector ensures `@ax/tool-dispatcher` inits first (it registers `tool:register`).
4. For `executesIn: 'host'` tools, the same plugin also registers `tool:execute:${name}` service hook.

### Catalog delivery

At sandbox startup, runner calls `ipcCall('tool.list')` → receives the full catalog → builds its internal dispatcher.

Cache-once-per-session is fine; tool set is immutable for a session's lifetime (invariant: no runtime tool registration). If future work needs dynamic tools, the runner polls or the host pushes via the inbox.

### Why not one-plugin-per-tool

`@ax/memory-cortex` will contribute 2–3 tools (store, recall, forget) plus subscriber hooks plus service hooks (internal). Splitting each tool into its own plugin means N manifests, N packages, N test suites for one conceptual unit. The `registers: ['tool:execute:memory.store', 'tool:execute:memory.recall', 'memory:record']` manifest list is the clean path — plugins group related capabilities.

## Workspace coordination contract

Full implementation is Week 7–9 (`@ax/workspace-git`). The **contract** ships in 6.5a via the IPC protocol so Week 7–9 consumers land against a stable shape.

### Flow

1. Agent in sandbox modifies files via tools. Sandbox-side git is local.
2. At turn end (or per-tool-call; runner-specific), sandbox commits locally with a synthetic commit message.
3. Sandbox sends `workspace.commit-notify` RPC with `{ parentVersion, commitRef, message }`.
4. Host's `@ax/workspace-git`:
   - Fetches the commit via git-http-backend from the sandbox's git dir (subprocess mode: bind-mount; k8s mode: sidecar).
   - Builds a `WorkspaceDelta` (lazy-content-fetcher shape per ax-conventions).
   - Fires `workspace:pre-apply` — subscriber chain. Any subscriber can reject.
5. If rejected: host returns `{ accepted: false, reason }`. Sandbox `git reset --hard parentVersion`; surfaces an error to the model on next turn.
6. If accepted: host promotes the commit to its authoritative store, fires `workspace:applied`, returns `{ accepted: true, version, delta }`.

### Subscribers that will hang off this contract (future slices)

- **Skill validator** (`workspace:pre-apply`) — rejects malformed skill files.
- **Canary** (`workspace:pre-apply`) — rejects commits containing secrets.
- **Skill indexer** (`workspace:applied`) — updates search index.
- **Audit** (`workspace:applied`) — logs the commit.

### Rewind semantics

The sandbox must be able to undo a rejected commit cleanly. `@ax/agent-runner-core` provides a `commitAndNotify(parentVersion, message)` helper that:
1. Does `git commit`.
2. Calls `workspace.commit-notify`.
3. On rejection: `git reset --hard parentVersion`; surfaces `WorkspaceRejectedError` to the runner's tool-loop.
4. On accept: returns the new `version` token.

Runners surface the rejection to the model as a tool error so the model can retry with corrections.

### 6.5a scope

No `@ax/workspace-git` ships. No subscribers wired. The `workspace.commit-notify` action exists in `@ax/ipc-protocol` schemas; the host's `@ax/ipc-server` returns `{ accepted: true, version: '<stub>', delta: null }` unconditionally. Agent's file edits work; no host-side diff awareness.

6.5a acceptance test does not cover workspace. Week 7–9 is when the contract earns its keep.

## Boundary review — the five invariants

### I1. Hook surface is transport- and storage-agnostic

- `ToolDescriptor.executesIn: 'sandbox' | 'host'` — closed set, no backend-specific values. ✅
- `WorkspaceVersion` — opaque token, no git vocabulary in payloads. ✅
- `session.next-message` — no postgres-specific fields; `cursor` is just a monotonic int. ✅
- `IPC transport` — abstracted behind `@ax/ipc-protocol`; unix socket vs cluster HTTP is a per-plugin implementation detail. ✅

### I2. No cross-plugin imports

All cross-plugin communication via hook bus (host-side) and IPC protocol (host↔sandbox). Only `@ax/core`, `@ax/ipc-protocol`, and `@ax/agent-runner-core` are shared imports.

The runner imports `@ax/agent-runner-core`; that's one-way. Runner does not import `@ax/tool-bash` (host-side plugin) — tool descriptors arrive over IPC; sandbox-local implementations live in separate `-impl` packages. ✅

### I3. No half-wired plugins

Every new 6.5a plugin is fully wired end-to-end in its shipping PR:

- `@ax/ipc-server` → called by `@ax/chat-orchestrator` → exercised in acceptance test.
- `@ax/session-inmemory` → called by `@ax/chat-orchestrator` and `@ax/ipc-server`.
- `@ax/agent-native-runner` → exercised in acceptance test.
- `@ax/llm-openrouter` → exercised in acceptance test (preset toggles to it).

No "wire this up later" packages. ✅

### I4. One source of truth per concept

- Tool catalog: `@ax/tool-dispatcher` only. Runners don't maintain their own registries.
- Session state: `@ax/session-*` only. Runner doesn't cache (reads from IPC).
- LLM credentials: host-side `@ax/llm-*` plugins. Sandbox never holds them. ✅

### I5. Capabilities explicit and minimized

- **Sandbox capabilities:** unix socket (or egress to one host URL), local filesystem scoped to workspace dir, process spawn for in-sandbox tool subprocesses. No direct DB, no cloud creds, no uncontrolled network egress.
- **IPC auth:** per-session bearer token, revocable, scoped to that session's IPC actions. Cannot impersonate another session.
- **Tool input validation:** every `tool:execute:${name}` handler validates input via Zod before execution. `tool.pre-call` subscriber chain can reject before the tool runs.
- **Untrusted content tagging:** LLM output and tool output are untrusted; `llm:pre-call` subscribers treat history content as untrusted.

**Security checklist required for every slice's PR.**
- **6.5a:** `@ax/ipc-server` is new attack surface; sandbox primitives cross the trust boundary.
- **6.5b:** `@ax/llm-openrouter` is a new third-party endpoint + new dep.
- **6.5c:** `@mariozechner/pi-coding-agent` and transitive deps land inside the sandbox.
- **6.5d:** `@anthropic-ai/claude-agent-sdk` spawns a subprocess tree inside the sandbox; `@ax/llm-proxy-anthropic-format` is new host-side HTTP surface.

Three threat models apply throughout: sandbox escape, prompt injection, supply chain.

## Scope decisions for the 6.5a implementation plan

These are 6.5a-specific open questions. Each needs an explicit call before code lands. Later slices (6.5b–d) surface their own questions when their plans get written; the list below is scoped to the topology shift.

1. **Runner binary packaging.** Single binary per runner (bundled with tool impls), or runner binary + tool impls as dlopen-able modules? Recommendation: single binary per runner. Simpler, ships in one dockerfile COPY line. Re-examine if the binary gets large enough to matter.

2. **Auth token scheme.** Opaque random string (64 bytes base64)? JWT (signed, carries claims)? Recommendation: opaque random, verified against session store on every IPC call. JWT adds signing ceremony without clear benefit since we have the store anyway.

3. **IPC request timeouts.** Per-action or global default? v1 uses 5min for `agent_response`, 15min safety net, per-call configurable. Recommendation: per-action defaults in `@ax/ipc-protocol` (e.g., `llm.call` = 5min, `tool.pre-call` = 10s, `session.next-message` = 30s long-poll), global override in config.

4. **Concurrent IPC from sandbox.** Can multiple RPCs be in flight at once (e.g., streaming LLM chunks while issuing tool.pre-call for a queued call)? Recommendation: yes, HTTP supports it natively; runner-core uses an HTTP keep-alive connection pool. Ordering of fire-and-forget events is best-effort.

5. **Cursor semantics for `session.next-message`.** Monotonic int per session? Recommendation: int; inbox entries carry their own index; cursor = last-seen-index. Sandbox advances only after processing, so crashes replay from the last-processed cursor.

6. **Eventbus in 6.5a?** Not needed for single-host/single-replica, but `chat:stream-chunk` fan-out wants it once multi-replica arrives. Recommendation: skip for 6.5a, ship alongside `@ax/eventbus-postgres` in Week 7–9. Document the fan-out path in the design so nothing surprises us later.

7. **How much of `chat-loop.ts` survives.** The turn-loop shape ports to the sandbox-side runner. The host-side orchestrator is thinner — it spawns the sandbox, waits for chat:end, returns outcome. Recommendation: delete `chat-loop.ts` as part of 6.5a; don't leave the dead code around.

8. **ax.config.ts preset wiring.** `@ax/preset-local-native-anthropic`, `@ax/preset-local-native-openrouter`? Or a more flexible config-driven preset? Recommendation: keep current ax.config.ts loader; introduce named presets as sugar later when the combinatorial cost justifies them.
