# Week 9.5 handoff — multi-tenant model (auth + agents + admin)

**For:** session starting after Week 7–9 (k8s deployment) lands.
**Previous slices:** Weeks 1–2, 3, 4–6, 6.5a, 6.5d, 6.5e (MCP client), 7–9 (k8s + postgres + workspace-git).
**Successors:** Week 10–12 (channels + observability) depends on this slice's primitives for multi-user auth.

This slice exists because MVP features 2 and 6 (admin-configurable MCP, personal/team agents) are not in the original architecture doc's Week plan. The current Week 10–12 scope assumes single-user; adding auth + agents + ACLs on top of channels in one slice balloons it past reviewability. Splitting isolates the tenant concern from the channels concern.

---

## Goal

Introduce user identity, agent entities, and per-agent ACLs — the primitives Week 10–12 channels will sit on top of. Ship a minimal admin UI so users and MCP configs can be managed before chat UI comes online.

Translation: **who** is the user, **what** agent are they talking to, and **how** is access enforced.

---

## Deliverables

- **`@ax/http-server`** — pulled forward from Week 10–12. Provides `http:register-route(method, path, handler)` hook. Auth middleware runs here. Channels (Week 10–12) will register against it; admin routes register against it now.
- **`@ax/auth`** — user identity. OAuth/OIDC provider integration (Google for MVP; OIDC-shaped config so any compliant IdP works later). Issues session cookies, resolves cookies to user identity. Exposes `auth:require-user(req) → User | 401` middleware hook consumed by `@ax/http-server`.
- **`@ax/agents`** — agent entity plugin. Postgres-backed. Agent = `{ id, owner_id, owner_type: 'user' | 'team', visibility: 'personal' | 'team', system_prompt, allowed_tools[], mcp_config_ids[], model, workspace_ref?, created_at, updated_at }`. Registers `agents:resolve(agent_id, user_id) → Agent | 'forbidden'` service hook that returns the agent IF the user has access, else rejects.
- **`@ax/teams`** *(optional for MVP, decide below)* — team entity + membership. Without it, agents with `owner_type: 'team'` are half-wired.
- **Chat-orchestrator extension** — `chat:run` gains `agent_id` and `user_id` params. Orchestrator calls `agents:resolve(agent_id, user_id)` before spawning a sandbox; passes agent config to `sandbox:open-session` (new payload fields: `systemPrompt`, `allowedTools`, `mcpConfigIds`, `model`).
- **Tool-dispatcher / MCP-client extension** — `tool:list` becomes agent-scoped. Dispatcher returns only tools in the agent's `allowed_tools` + MCP tools from its `mcp_config_ids`. This is the per-agent scoping deferred from 6.5e.
- **Admin API routes** — mount on `@ax/http-server` under `/admin/*`:
  - `POST /admin/agents`, `GET /admin/agents`, `PATCH /admin/agents/:id`, `DELETE /admin/agents/:id`
  - `POST /admin/mcp-servers`, `GET /admin/mcp-servers`, `PATCH /admin/mcp-servers/:id`, `DELETE /admin/mcp-servers/:id` (moved from 6.5e's placeholder surface)
  - `POST /admin/mcp-servers/:id/test`
  - If `@ax/teams` ships: `POST /admin/teams`, team-membership endpoints.
  - All require authenticated session + authz check.
- **Bootstrap CLI** — `ax admin bootstrap` (or similar). Reads initial-admin credentials from env / a one-time bootstrap token and calls `/admin/agents` etc. to create the first user, agent, and (optional) MCP config before any UI exists. Needed because 9.5 ships before the chat UI in Week 10–12 — nothing else has a way to drive the admin API yet. Tests also use it for fixture setup.

## Scope decisions to make while writing the plan

1. **Auth provider set.** MVP recommendation: Google OAuth via OIDC, plus a dev-mode bootstrap (env-var admin token) for local development. Full username/password auth is deferred — OAuth is cheaper to get right. Generic OIDC shape means GitHub/Okta/etc. drop in later without plugin changes.

2. **Visibility model.** Options:
   - **(a)** `personal` and `team` only.
   - **(b)** `personal`, `team`, `public` (anyone signed in can use).
   - **Recommendation:** (a) for MVP. `public` invites sharing/abuse concerns that aren't urgent. Add in Week 13+ if a real use case surfaces.

3. **Where ACL logic lives.**
   - **(a)** `@ax/agents` plugin owns ACL checks inline (`agents:resolve` is the authz gate).
   - **(b)** Separate `@ax/authz` plugin with a generic `authz:check(subject, action, resource)` hook.
   - **Recommendation:** (a). Authz for this MVP is simple: agent owner_id matches user, or user is team member. One plugin, one check. Extract to `@ax/authz` only when a second resource type (e.g., shared MCP configs that aren't tied to an agent) needs the same pattern.

4. **Admin surface: no separate UI — merged into the chat UI's user menu in Week 10–12.** The admin operations (agent CRUD, MCP server config, team management if it ships) show up as additional user-menu entries inside the same chat UI users already know. Implications for 9.5:
   - **9.5 ships:** admin HTTP endpoints + auth middleware + `ax admin bootstrap` CLI for creating the initial admin before any UI exists. Acceptance tests drive the endpoints via curl / fetch; no UI to click.
   - **10–12 adds:** user-menu entries that call the same endpoints, plus the forms/modals to drive them.
   - **Why one UI:** single auth flow, single design language, contextual placement ("my agents" lives in the same menu as the chat a user is having).
   - **Authz must be at the API layer, not UI.** Hiding an admin menu entry is not access control — the endpoints enforce permissions regardless of who calls them. Flag this explicitly in the 10–12 plan so the chat UI team doesn't conflate "hidden from the dropdown" with "secured."

5. **Teams primitive scope.**
   - **(a)** Ship `@ax/teams` in this slice. Agents with `owner_type: 'team'` are fully wired.
   - **(b)** Defer teams. MVP ships with only `personal` agents; "team agent" is a post-MVP feature.
   - **Recommendation:** (a) if feature 6 ("shared team agents") is truly MVP. Otherwise (b). Check with the user before committing — teams is a real chunk of work (membership, invites, role model).

6. **Session + agent wiring.**
   - `@ax/session-postgres` sessions gain `user_id` and `agent_id` columns. Migration owned by `@ax/session-postgres` (not `@ax/agents` — sessions are the session plugin's concern).
   - `chat:run(user_id, agent_id, message)`. Orchestrator's resolved agent config flows into `sandbox:open-session`.
   - Runner inside the sandbox reads agent config (from session-resolve response or IPC `session.config` action) and configures the claude-sdk with system_prompt / allowedTools / mcpServers.

7. **Per-agent MCP scoping details.** When `@ax/mcp-client` ships (6.5e) it registers all configured MCP tools globally. In 9.5, `@ax/tool-dispatcher` filters `tool:list` responses by the calling session's agent. MCP client itself doesn't need changes; the filter is at the dispatcher.

8. **Workspace ownership.** Agents may reference a workspace (`workspace_ref`). For personal agents: private workspace per agent. For team agents: shared workspace. `@ax/workspace-git` needs to honor workspace scoping by agent, not global. If Week 7–9 shipped workspace as globally-scoped, this slice extends it; if already agent-scoped, no change. Verify during plan-writing.

9. **Session-to-agent immutability.** Can a running session's `agent_id` change mid-chat? Recommendation: no. A session is bound to one agent at creation. Switching agents = new session. Simplifies authz and tool-catalog caching.

## Boundary review

- **Hook payload field names.** `user_id`, `agent_id`, `owner_type` — generic, not backend-specific. ✅ `oauth_subject_id` in `@ax/auth` is IdP-shaped; rename to `auth_subject_id` with the IdP name in a separate field. ✅
- **`agents:resolve` service hook** — closed verdict set (`'forbidden'` vs resolved agent). No data leak on rejection beyond "forbidden". ✅
- **Alternative impl for `@ax/auth`** — another OIDC provider, SAML, or local password. All can register `auth:require-user`. ✅
- **Alternative impl for `@ax/agents`** — no obvious one today, but a future "git-backed agent definitions" plugin (agents stored as files in a repo) would implement the same service hooks. Contract holds.
- **Subscriber risk.** Week 10–12's audit plugin will subscribe to auth events. Make sure `auth:user-signed-in(user_id)` payload is generic.

## Security — `security-checklist` required (heavy)

This is the biggest trust-boundary slice since 6.5a. Three threat models all apply:

- **Supply chain:** OAuth library, OIDC discovery library, CSRF protection library. Pin versions tightly.
- **Prompt injection:** Agent `system_prompt` is user-authored. Personal agents = user prompts their own agent; fine. But **team agents mean one user writes prompts another user runs** — that's prompt injection across trust boundary. Mitigations: display system_prompt prominently before a user starts a chat with a team agent; warn on changes; require confirmation.
- **Authz escalation:** Every postgres query from a logged-in user MUST filter by user/team. A missing `WHERE` clause leaks other tenants' data. Enforce via a query-building helper that takes a required `scope` param; bare `SELECT * FROM agents` should be disallowed by convention and lint rule.

Additional concerns:
- **Session cookie handling.** HttpOnly, Secure, SameSite=Lax. Short lifetime + refresh.
- **CSRF on admin API.** Double-submit cookie or origin check.
- **Rate limiting on auth endpoints.** Brute-force protection.
- **Audit trail for admin actions.** `@ax/audit` (Week 10–12) will subscribe; for 9.5 at minimum log via the existing structured logger.
- **MCP config secrets.** Credentials for MCP servers must flow through `@ax/credentials` (from 6.5e); the admin API never echoes decrypted values.
- **Cross-tenant MCP leakage.** User A's MCP config must never appear in User B's tool list. Per-agent filter in `@ax/tool-dispatcher` is the enforcement point; test for it explicitly.

## Acceptance test for Week 9.5

Because there's no UI yet (that lands in Week 10–12), all tests drive the admin API directly via HTTP.

Automated:
1. `ax admin bootstrap` creates the initial admin user from env credentials. Subsequent admin endpoints accept the resulting session.
2. User A signs in via OAuth, calls `POST /admin/agents` to create a personal agent. Chat works end-to-end with the new agent-scoped session flow.
3. User B signs in. `GET /admin/agents` returns User B's agents only; User A's are invisible.
4. User B attempts `POST /chat` with User A's agent_id → 403.
5. Team admin creates a team agent, adds User A to the team. User A can now `GET /admin/agents` and see the team agent + chat against it.
6. User A configures an MCP server for their personal agent. Tool-list for User A's agent session includes MCP tools; User B's agent session does not.
7. Unauthenticated request to any `/admin/*` endpoint → 401.
8. Session fixation / CSRF / cookie theft attempts (structured pen-test — pick 3 classic attacks, verify hardening).

Manual: end-to-end OAuth sign-in flow with a real Google account; bootstrap admin; create an agent and MCP config via curl; run a chat against the resulting agent via the existing CLI.

## Out of scope / deferred

- Role-based access within a team (admin vs member) → Week 13+ if teams outgrow flat membership.
- Public agents (anyone-signed-in access) → Week 13+.
- SAML, LDAP → Week 13+.
- SSO session revocation beyond cookie expiration → Week 13+.
- Audit log of admin actions as a dedicated audit trail → Week 10–12's `@ax/audit` absorbs this.
- Fine-grained workspace ACLs (per-file, per-path) → not in roadmap.
- Agent definitions as code/config (git-backed) → Week 13+.
