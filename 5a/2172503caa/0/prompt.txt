### Deviations from plan

1. Plan uses `{ path: ':memory:' }` for `createStorageSqlitePlugin`; actual config field is `databasePath`. Used the real field everywhere.
2. Plan tests use `.rejects.toThrow(/credential-not-found/)` ŌĆö `.toThrow()` matches against `.message`, but the error's plain message is `"no credential for ref='X'"`. Used `.rejects.toMatchObject({ code: 'credential-not-found' })` to mirror the existing `plugin.test.ts` pattern.
3. Plan's storage-sqlite snippet uses raw `db.prepare(...).all(...)`; actual storage-sqlite uses Kysely. Implemented via `selectFrom('kv').where(sql\`key LIKE ${...} ESCAPE '\\\\'\`)` with explicit `%`/`_`/`\\` escaping. Mirrored in storage-postgres.
4. Updated CLI to pass `scope:'user', ownerId:CLI_USER_ID` in **Task 1.3** (not 1.6), because the facade interface change otherwise breaks every CLI test immediately. The `migrate` subcommand still landed in 1.6 per plan.
5. Per plan Task 1.4's note about the inflight mutex keyed on `(userId, ref)`: kept the existing key shape ŌĆö the precedence walk happens inside `doResolve`, the mutex correctly serializes any concurrent `credentials:get(ref, userId)` calls regardless of which scope they end up resolving from.
6. **Behavior change: tombstone-vs-envFallback** (intentional, documented at the design level by the plan's `if (env.isTombstone) continue;` semantics). Pre-precedence: deleting a user-scoped credential made `credentials:get` reject with `credential-not-found` even if `envFallback` had a mapping. Post-precedence: a user-scope tombstone falls through to agent ŌåÆ global ŌåÆ envFallback. Operators relying on "delete means deny everywhere" should leave envFallback empty. Existing `plugin.test.ts` test (`envFallback does NOT override a tombstoned credential`) was rewritten to assert the new contract (`envFallback fires when every scope is tombstoned`). Worth flagging in PR notes.
