Documentation

Install & setup, in plain language.

ax-next is a tiny kernel and a lot of plugins. This guide gets you from a clean machine to a real agent answering real questions — first on your laptop, then on Kubernetes if you need it.

You'll need
Node ≥ 24, pnpm ≥ 10
Local time
~10 min
Kubernetes time
~20 min

01 Prerequisites

A short list of things to have on hand before starting.

Node and pnpm

ax-next pins node ≥ 24 and pnpm ≥ 10 in package.json. Older versions install but pnpm warns loudly and we don't test against them. Better to be on a current Node.

verifyshell
$ node --version
v24.4.0
$ pnpm --version
10.23.0

Anthropic API key (optional, for now)

Without one, ax-next runs in mock mode and replies with a canned "hello". With one, you get a real agent that can run bash and read or write files. Get a key from console.anthropic.com and export it as ANTHROPIC_API_KEY.

Note

The API key is real money once requests start flowing. Don't paste it into a chat window, don't commit it, don't share it in a screenshot.

For Kubernetes only

If you intend to deploy to a cluster, you'll also need:

  • Docker — to build the runner image
  • kind — a local Kubernetes-in-Docker cluster
  • Helm 3 — to install the chart

02 Install & build

Same three commands whether you're heading toward local or Kubernetes.

The build uses TypeScript project references and walks the package graph in the right order automatically.

installshell
$ git clone https://github.com/project-ax/ax-next.git
$ cd ax-next

$ pnpm install
$ pnpm build

# Optional — fast, finishes in seconds
$ pnpm test
Workspace ordering

If you later filter to a single package and it complains about @ax/core, build core first: pnpm --filter @ax/core build. Running pnpm build at the root handles ordering for you.

03 First run

Confirm it works before spending real tokens.

With no ax.config.ts in the current directory, the CLI falls back to defaults — llm: 'mock', no tools, no real calls. This is the smoke test.

smoke testshell
$ ./packages/cli/dist/main.js "hello there"
hello

That's it. If you got hello back, the kernel booted, the mock LLM registered, the chat loop ran, and the audit-log subscriber wrote a row to SQLite. Exit codes: 0 on success, 1 on terminated, 2 on fatal error.

04 Local — Configure

Swap the mock for the real Anthropic API.

Drop an ax.config.ts in your working directory. The CLI looks in the cwd by default.

ax.config.tstypescript
export default {
  llm: 'anthropic',
  tools: ['bash', 'file-io'],
  anthropic: { model: 'claude-sonnet-4-6' },
};

Each string in tools turns on a plugin: bash runs commands through the subprocess sandbox, file-io exposes path-checked read_file and write_file.

You then have two ways to authenticate with Anthropic. Pick one:

  • API key — fastest if you already have one. Export ANTHROPIC_API_KEY and skip to §06.
  • Claude Pro / Max account via OAuth — sign in with your existing subscription instead of paying per token. See §05.

05 Local — Sign in with Claude Pro / Max

PKCE OAuth against your existing Anthropic subscription.

If you have a Claude Pro or Max subscription, you can use it to power ax-next directly — no separate API spend. The CLI runs a standard public-client PKCE flow: it opens a browser, you sign in to your Anthropic account, and the resulting tokens are stored locally, encrypted at rest.

Set the encryption key

OAuth tokens are encrypted at rest with AES-256-GCM. The key comes from AX_CREDENTIALS_KEY — required, 32 bytes, base64. Generate it once and stash it somewhere durable; if you lose it, you'll need to re-run login.

one-time setupshell
$ export AX_CREDENTIALS_KEY=$(openssl rand -base64 32)

# Persist it in your shell profile so future sessions can decrypt
# the stored OAuth blob. Treat this value like a database password.

Sign in

Run credentials login anthropic. The default credential ref is anthropic-personal; pass a different one if you want to keep multiple accounts side by side.

loginshell
$ ./packages/cli/dist/main.js credentials login anthropic

# Or, with a custom ref:
$ ./packages/cli/dist/main.js credentials login anthropic work-acct

What happens next:

  1. The CLI binds 127.0.0.1:1455 to receive the redirect.
  2. Your default browser opens claude.ai/oauth/authorize. Sign in with the Anthropic account that has your Pro / Max plan.
  3. Approve the authorization. Anthropic redirects back to the local listener; the CLI matches the CSRF state, exchanges the code for tokens, and stores them under the chosen ref with kind anthropic-oauth.
  4. The browser tab shows a confirmation; the CLI prints the stored ref and exits.
Refresh is automatic

Anthropic access tokens last about an hour. ax-next refreshes them 5 minutes before they expire, transparently — you sign in once and don't think about it again.

Point the LLM plugin at the OAuth credential

Tell llm:anthropic to resolve the credential ref instead of reading ANTHROPIC_API_KEY:

ax.config.tstypescript
export default {
  llm: 'anthropic',
  tools: ['bash', 'file-io'],
  anthropic: {
    model: 'claude-sonnet-4-6',
    credentialRef: 'anthropic-personal',
  },
};

Now ./packages/cli/dist/main.js "..." uses your Pro / Max subscription. No ANTHROPIC_API_KEY needed.

Field note

The login listener has a 60-second timeout. If your browser flow stalls, just re-run the command. The PKCE verifier is regenerated each time, so there's nothing to clean up.

If localhost resolves to ::1 on your system and the redirect hangs, the listener is bound to 127.0.0.1 on purpose (RFC 8252 loopback). Use the IPv4 literal — it's already what the redirect URI specifies.

06 Local — Run a real turn

runshell
$ export ANTHROPIC_API_KEY=sk-ant-...
$ ./packages/cli/dist/main.js "list the files in this directory"

The CLI fires chat:start, routes through llm:pre-call → llm:call → llm:post-call, dispatches any tool calls through tool:execute → tool:execute:<name> → sandbox:spawn, loops until the model stops asking for tools, fires chat:end, and the audit-log plugin writes the outcome to SQLite.

07 Local — Where state lives

By default, the audit log lands in ./ax-next-chat.sqlite next to your config. Override with AX_DB or set storageSqlite.databasePath in the config.

relocateshell
$ export AX_DB=/tmp/ax.sqlite
$ ./packages/cli/dist/main.js "how many tokens did I just spend"

08 Local — MCP servers

Plug in external tools through Model Context Protocol.

Want your agent to reach Notion, a filesystem indexer, an internal API? MCP servers plug in via @ax/mcp-client with two commands.

registershell
$ ax-next mcp add my-server \
    --transport stdio \
    --command "node /path/to/server.js"

$ ax-next credentials set my-server.token "sk-..."

Tools surface to the model as mcp.<server>.<tool> and fire the same tool:pre-call / tool:post-call hooks as built-ins, so audit and observability work uniformly.

Trust boundary

MCP servers run inside your trust zone. Treat their tool descriptions and output as untrusted at every hop — the model sees them, and a compromised server can attempt prompt injection.

09 Kubernetes — Spin up kind

The local-cluster path is the canary path.

The Helm chart lives at deploy/charts/ax-next/. It ships a single-replica host Deployment, scoped RBAC, two NetworkPolicies, and an optional embedded Postgres with pgvector.

One pod runs the host (kernel + plugins). When the agent calls a tool, the host asks Kubernetes to spawn a short-lived runner pod in a separate namespace. Network policies fence the runner so it can only talk to the host. The host can only talk to Postgres, the k8s API, DNS, and HTTPS. Everything else is denied.

setupshell
# 1. Spin up a cluster
$ kind create cluster --name ax-next-dev

# 2. Build and load the runner image
$ docker build -t ax-next/agent:dev -f deploy/Dockerfile.agent .
$ kind load docker-image ax-next/agent:dev --name ax-next-dev

# 3. Pull subchart deps (Postgres lives there)
$ helm dependency update deploy/charts/ax-next

# 4. Create the runner namespace — the chart does not do this for you
$ kubectl create namespace ax-next-runners

10 Kubernetes — Install the chart

helm installshell
$ helm install ax-next deploy/charts/ax-next \
    -f deploy/charts/ax-next/kind-dev-values.yaml \
    --set credentials.key=$(openssl rand -base64 32) \
    --set anthropic.apiKey=$ANTHROPIC_API_KEY

$ kubectl rollout status deployment/ax-next-host
$ kubectl rollout status statefulset/ax-next-postgresql

$ kubectl port-forward svc/ax-next-host 8080:80

The host is reachable on localhost:8080. Smoke-test with curl against the health endpoint until the web channel lands.

Lint and preview

Before installing against a real cluster, render the manifest and look at it. RBAC, NetworkPolicies, and namespace fences are easier to sanity-check as YAML than as a running incident.

previewshell
$ helm lint deploy/charts/ax-next

$ helm template ax-next deploy/charts/ax-next \
    -f deploy/charts/ax-next/kind-dev-values.yaml \
    --set credentials.key=dGVzdA== --set anthropic.apiKey=test \
    | kubeconform -strict

11 Kubernetes — Sign in with Claude Pro / Max

Bridging the pod's OAuth listener to your laptop's browser.

When ax-next runs in a pod, the OAuth listener and your browser don't share a loopback. The redirect URI http://127.0.0.1:1455/callback is hardcoded and Anthropic-whitelisted — we can't move it to a different host or port. So we bridge the pod's 127.0.0.1:1455 to your laptop's 127.0.0.1:1455 for the duration of the login.

Step 1 · Port-forward the callback

In one terminal, forward the OAuth callback port from the host pod to your laptop. Leave it running for the rest of the flow.

terminal Ashell
$ kubectl port-forward deploy/ax-next-host 1455:1455

Step 2 · Run the login from inside the pod

In a second terminal, exec into the host pod and start the login flow. The chart already injects AX_CREDENTIALS_KEY from the chart Secret, so the encrypt-at-rest path is wired.

terminal Bshell
$ kubectl exec -it deploy/ax-next-host -- \
    ax-next credentials login anthropic

# Or with a custom ref:
$ kubectl exec -it deploy/ax-next-host -- \
    ax-next credentials login anthropic team-shared

Step 3 · Sign in from your laptop's browser

The CLI prints an authorize URL. Open it in your browser, sign in with the Anthropic account that has your Pro / Max plan, and approve the authorization.

The browser redirects to http://127.0.0.1:1455/callback. Because of Step 1, that hits your laptop's port-forward, which forwards to the pod's 127.0.0.1:1455, where the CLI's listener is waiting. State is verified, the code is exchanged for tokens, and the encrypted blob is written to the in-cluster Postgres under the chosen ref.

60-second window

The listener times out after a minute. If your browser flow stalls, stop the port-forward, re-run the kubectl exec command, and try again. Each attempt regenerates the PKCE verifier, so there's no stale state to clean up.

Step 4 · Point the host at the credential

Set anthropic.credentialRef in the chart values so the host's llm:anthropic plugin resolves the OAuth credential instead of looking for an API key. anthropic.apiKey can be left unset.

values.yamlyaml
anthropic:
  model: claude-sonnet-4-6
  credentialRef: anthropic-personal

Apply with helm upgrade (passing the same credentials.key as the install — see §13). Refresh is automatic from there: tokens roll over 5 minutes before expiry without operator involvement.

Heads up · web UI flow is planned, not shipped

The web channel (Phase 10–12) will expose HTTP routes — POST /auth/anthropic/login & /auth/anthropic/callback — that call the same credentials:login:anthropic-oauth and credentials:exchange:anthropic-oauth sub-services as the CLI. Same PKCE flow, no port-forward needed, signed in straight from the browser. Until that lands, kubectl exec + port-forward is the path. We'll update this section when it ships.

12 Kubernetes — Iterate on changes

rebuild & reloadshell
$ docker build -t ax-next/agent:dev -f deploy/Dockerfile.agent .
$ kind load docker-image ax-next/agent:dev --name ax-next-dev
$ kubectl rollout restart deployment/ax-next-host

What you're agreeing to

  • The host pod gets pod-CRUD verbs in ax-next-runners. If that namespace doesn't enforce pod-security.kubernetes.io/enforce: restricted, a misconfigured workload could blast wider than the chart can fence.
  • Runner pods default to runtimeClassName: gvisor. Without gVisor on the cluster, pod creates fail loudly with runtimeClassName not found.
  • NetworkPolicies need a CNI that enforces them (Calico, Cilium, native). Plain kind doesn't enforce — the kind-dev values disable the policies on purpose.

13 Kubernetes — Credentials key

Read this before running helm upgrade.

Important

AX_CREDENTIALS_KEY encrypts every credential at rest. If you change the key, every credential encrypted with the old one becomes unrecoverable. There is no recovery path.

Stash the key somewhere durable on day one (sealed-secret, vault, external secrets manager — whatever you trust), and pass the same value to every helm upgrade. Re-running openssl rand -base64 32 on every upgrade silently bricks all stored credentials.

The chart has a guard against in-cluster overwrites, but GitOps tools that render offline (Argo CD, Flux HelmRelease, plain helm template | kubectl apply) bypass it. Plan accordingly.

14 Configuration reference

Environment variables

VariablePurpose
ANTHROPIC_API_KEY API key for the llm:anthropic plugin. Required when llm: 'anthropic'.
AX_DB Path to the SQLite file used by @ax/audit-log. Defaults to ./ax-next-chat.sqlite.
AX_CREDENTIALS_KEY Base64 32-byte key for AES-256-GCM credential encryption. Required for credential plugins. Do not lose.
AX_CONFIG Override the path to ax.config.ts. Defaults to cwd.

ax.config.ts shape

FieldDescription
llm'mock' | 'anthropic'. The provider plugin to load.
toolsArray of tool names. Today: 'bash', 'file-io'.
anthropic.modelModel id, e.g. 'claude-sonnet-4-6'.
anthropic.credentialRef Resolve auth from a stored credential (e.g. 'anthropic-personal') instead of ANTHROPIC_API_KEY. Use this to sign in with a Claude Pro / Max account via OAuth.
storageSqlite.databasePathOverride the SQLite path without touching env.

15 Troubleshooting

pnpm install complains about Node

You're on Node < 24. Use nvm install 24 or your version manager of choice.

CLI replies "hello" when you wanted real answers

You're in mock mode. Either there's no ax.config.ts, or its llm field is set to 'mock'. Set it to 'anthropic' and export ANTHROPIC_API_KEY.

Helm fails on runtimeClassName not found

Your cluster doesn't have gVisor and the chart defaults to runtimeClassName: gvisor. Either install gVisor (recommended) or override with --set runner.runtimeClassName="".

NetworkPolicies aren't fencing anything

Your CNI doesn't enforce them. Plain kind clusters don't. Try Calico, Cilium, or a managed cluster's native CNI.
