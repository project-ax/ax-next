| 2026-04-28 | Phase 1b kept consumer-facing `credentials:*` hooks unchanged; reshape deferred to Phase 3 | Section 7 of the agent-centric design called Phase 1b "mechanical refactor, behavior identical" while Section 3 specified a redesigned shape — internally inconsistent. | 2026-04-28 | `@ax/credentials-store-db` (Phase 1b) delegates to `storage:get` / `storage:set` rather than owning its own table + migrations | Design Section 3 says "owns the credentials table + Kysely migrations, dialect-agnostic via `database:get-instance`" — but `@ax/database-sqlite` doesn't exist (only postgres has the hook), and the dedicated table earns its weight only with Phase 3 columns (`kind`, `expiresAt`, `metadata`). The seam is still real: vault / KMS backends register `credentials:store-blob:*` against a different store. | Build `@ax/database-sqlite` first (rejected: bigger than Phase 1b); have store-db open its own better-sqlite3 connection (rejected: deferred — wait until Phase 3 schema needs it) |
| 2026-04-29 | Phase A spike confirms D1 (HOME-redirect): runner-owned-sessions design moves forward unchanged | Set HOME=tempdir, ran `query()` one-shot then `query({resume})` — sessionId reattached, magic phrase recalled verbatim. Two follow-ups for Phase C: (1) workspace persistence ignore-list keeps only `.claude/projects/`, drops `.claude.json` + backups + policy-limits; (2) `runner:read-transcript` uses sessionId glob, not encoded-cwd reconstruction (SDK uses `realpath(cwd)`, breaks on macOS `/var` → `/private/var`). | Bind-mount `~/.claude/projects/<cwd>` to a workspace path (rejected: more sandbox-specific plumbing, doesn't generalize to k8s — see design D1) |
| 2026-04-29 | No v1→v2 schema migrations in ax-next, ever | ax-next is greenfield with no production data to preserve. The "forward-only via future v2 side-table, never an in-place ALTER" rule in `conversations_v1` was inherited from a different mental model. ALTER TABLE in place is correct now AND for every future schema change. The `_v1` suffix on existing tables (`conversations_v1_*`, `audit_v1_*`, etc.) is now just a stable identifier, not a version pointer. Confirmed by Vinay 2026-04-29 during Phase B planning. | Maintain v1 → v2 split for breaking changes (rejected: greenfield, no rows to migrate; speculative versioning earns nothing) |
| 2026-05-21 | Finding 3 facade lives in `@ax/core` (`registerWorkspaceApplyFacade`); backends rename their raw hook to `workspace:apply-internal`; callers of `workspace:apply` are untouched | Facade logic (fire pre-apply → call internal → fire applied) is backend-agnostic, so core is the principled single home (a future GCS backend reuses it). Renaming the *registration* (not the call) means every existing `call('workspace:apply')` site routes through the facade with zero churn — the cleanest way to make the policy chokepoint unbypassable. `filterToPolicy` relocates to core (only depends on the `FileChange` core type) so the facade can policy-filter pre-apply identically to commit-notify. Full design: `docs/plans/2026-05-21-finding3-workspace-apply-facade.md`. | Per-backend duplicated facade (rejected: violates one-source-of-truth); facade in `@ax/workspace-git-core` (rejected: not backend-agnostic — GCS won't import it); also route `apply-bundle`/commit-notify through the facade (rejected: commit-notify already fires both hooks, not a bypasser; out of finding-3 scope) |
| 2026-05-21 | New `yolo-ship` skill is an **orchestrator that sequences existing skills**, not a re-implementation | The brainstorm/design/implement/ship phases map 1:1 onto superpowers:brainstorming / writing-plans / subagent-driven-development / finishing-a-development-branch + commit-push-pr. The skill adds only the autonomy contract + the CodeRabbit/CI ship loop, and cross-references the rest by name (no @ links, per writing-skills). | Write a self-contained monolith re-explaining each phase (rejected: duplicates skills that already exist + evolve independently; bloats context) |
| 2026-05-21 | `yolo-ship` keeps context low via **subagent-driven implementation + Explore for reads**; orchestrator holds only plan + memory + summaries | User's hard cap is < 300–400K tokens across all three phases. Per-task implementation subagents give ~50–100× savings; codebase exploration goes through Explore so file dumps never hit the orchestrator. | One long-running orchestrator agent doing everything inline (rejected: just relocates the context blowup); fully delegate brainstorm+design to a subagent too (rejected: orchestrator's judgment + memory are most valuable in those phases) |
| 2026-05-21 | `yolo-ship` runs brainstorming in **self-answering mode** and logs decisions instead of asking the user | User instruction: "don't ask questions unless absolutely necessary — trust your recommendations as long as you document them in memory." Brainstorming is normally interactive; here the orchestrator generates the questions, answers from code/memory/conventions, and records each in decisions.md. AskUserQuestion reserved for high-stakes + ambiguous + non-inferable. | Keep brainstorming interactive (rejected: contradicts the autonomy instruction) |
| 2026-05-21 | `yolo-ship` is a **project skill** (`.claude/skills/`, committed) and **`done = green review-clean PR, NOT merged`** | It wires into this repo's CodeRabbit + TODO.md + `.claude/memory/` conventions, so it belongs with the other day-1 project skills, not `~/.claude/skills`. Stop condition mirrors the user's spec exactly ("if everything looks good… you are done") — no auto-merge. | Personal skill in `~/.claude/skills` (rejected: CodeRabbit/TODO/memory wiring is repo-specific); auto-merge on green (rejected: user said stop at done) |
| 2026-05-21 | UI-bugs fix: implement the 5 small edits **inline** rather than via per-task subagents | yolo-ship mandates subagent-driven impl to protect orchestrator context, but the rationale is heavy exploration tasks. I already did the deep reads (root causes confirmed; ~5 surgical files). Spawning subagents to re-derive known context would cost MORE total tokens and risk divergence. Context budget stays well under the 300–400K cap. | Strict one-subagent-per-task (rejected here: overhead/divergence risk for tiny, fully-specified diffs) |
| 2026-05-21 | Title bug: replace the one-shot `assistantTurnCount !== 1` gate with a **bounded retry** (`<= MAX_TITLE_ATTEMPT_TURNS`, =3) while untitled | Root cause of "inconsistent" titles is the single attempt: if it's skipped/fails for ANY transient reason (jsonl-read race in `conversations:get`, llm:call error, validation rejection) the chat is permanently untitled. `title !== null` guard + `ifNull:true` on set-title already make re-attempts idempotent, so retrying on the next few assistant turns is safe; the cap bounds title-LLM spend if a model keeps emitting unusable titles. | Keep one-shot + only fix the UI (rejected: leaves "never generated" class unfixed); unbounded retry (rejected: unbounded LLM spend on degenerate output) |
| 2026-05-21 | Title bug (UI half): widen `generateTitle` poll window (more attempts + backoff, ~12-15s) instead of a client-side first-message fallback title | The 3×1s poll often gives up before the backend round-trip (jsonl read + haiku) lands, permanently caching "New Chat" until a full reload. Widening catches the common case. A client fallback title (first-message snippet) was rejected: it would later flip to the LLM title on reload — jarring. Residual (title landing after the widened window still needs a list() refresh) tracked as TODO. | Client-side fallback title (rejected: diverges from backend, flips on reload); push/websocket title update (rejected: out of scope for MVP) |
| 2026-05-21 | Attachment chip bug: **reconstruct the `attachment` block from the stored text-mention on the host read path** (`conversations:get`), gated to the conversation's own `.ax/uploads/<conversationId>/` prefix | Under runner-owned-sessions the jsonl stores the runner's *translation* (text-mention `User attached '<name>' at <path> (<mime>)`), not the original `attachment` block — so the original is genuinely gone. The mention embeds path+name+mime (everything the chip needs; `sizeBytes` is dropped by the render path anyway). Reconstruction needs no schema/table and **retroactively fixes already-broken chats** (no migration). Format owned by a shared `formatAttachmentMention`/`parseAttachmentMention` pair in `@ax/ipc-protocol` so producer (runner) and consumer (conversations) can't drift. Path-prefix gate + `AttachmentBlockSchema.safeParse` + role==='user'-only keep untrusted jsonl from injecting an out-of-scope downloadable ref. | Host-side per-turn attachment-block table overlaid on read (rejected: new table+hook+turnId-matching, heavier, and needs a migration to fix old chats; `sizeBytes` is the only extra fidelity and it's unused by the render path); client-side parse in blocksToParts (rejected: leaves the wire shape lossy for all other consumers of conversations:get) |
