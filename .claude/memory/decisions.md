# Decisions

Architectural / process decisions. Never deleted — strikethrough if reversed.

| Date | Decision | Rationale | Alternatives |
|---|---|---|---|
| 2026-04-23 | Ship `storage:get` / `storage:set` without `database:get-instance` abstraction | YAGNI until a second backend exists; architecture doc Section 6 defers this until `@ax/storage-postgres` lands | Introduce the abstraction now (rejected: premature, no second impl to validate shape) |
| 2026-04-23 | Ship `storage:get` without a runtime consumer in Week 3 | Symmetric API with `storage:set`; consumer (session lookup) lands Week 4+. Documented waiver in PR #2 | Wait until Week 4 to register `storage:get` (rejected: half-wired feels worse than a one-slice-early symmetric pair) |
| 2026-04-23 | `@ax/audit-log` owns the `chat:<reqId>` key namespace in storage | One-source-of-truth invariant — any other plugin wanting to read audit records does so through a future service hook, not by reading the key directly | Shared key convention across plugins (rejected: violates invariant #4) |
| 2026-04-28 | Phase 1b kept consumer-facing `credentials:*` hooks unchanged; reshape deferred to Phase 3 | Section 7 of the agent-centric design called Phase 1b "mechanical refactor, behavior identical" while Section 3 specified a redesigned shape — internally inconsistent. Picked the conservative reading: ship the seam (`credentials:store-blob:*`) without churning all call sites with no per-kind benefit yet. Phase 3 reshapes when `credentials:resolve:anthropic-oauth` actually needs `kind`. | Reshape now per Section 3 (rejected: ~600 LOC of churn across credential-proxy + mcp-client + CLI for no Phase 1b benefit) |
| 2026-04-28 | `@ax/credentials-store-db` (Phase 1b) delegates to `storage:get` / `storage:set` rather than owning its own table + migrations | Design Section 3 says "owns the credentials table + Kysely migrations, dialect-agnostic via `database:get-instance`" — but `@ax/database-sqlite` doesn't exist (only postgres has the hook), and the dedicated table earns its weight only with Phase 3 columns (`kind`, `expiresAt`, `metadata`). The seam is still real: vault / KMS backends register `credentials:store-blob:*` against a different store. | Build `@ax/database-sqlite` first (rejected: bigger than Phase 1b); have store-db open its own better-sqlite3 connection (rejected: deferred — wait until Phase 3 schema needs it) |
