## Purge-on-delete: gate the dangerous primitive on caller authority, not entry routes (2026-06-01)

When a delete cascades into deleting OTHER state (here: `connectors:delete` purging the
connector's vault keys), gate the dangerous part at the single primitive every path funnels
through, not at one entry route. Gating only the HTTP `/settings/connectors` keyMode left the
authored-connector approve path (`connector_propose` → `connectors:install-authored` →
`applyAuthoredConnectorGrant` promotes via the `connectors:upsert` HOOK) as a second non-admin
door to the same cross-tenant DoS.

Pattern: the hook takes an explicit caller-granted authority flag (`purgeGlobal?: boolean`,
default false / fail-safe), the route passes `actor.isAdmin`, and the hook skips the
shared/global-scope deletion unless authorized. This is invariant-#1-clean (a granted-capability
boolean, no auth/backend vocabulary in the hook) and closes the CLASS of bug. A GLOBAL-scope
credential is owner-independent (keyed by connector id), so it's shared infra — only an admin may
tombstone it; a non-admin's delete still purges their own `scope:'user'` refs.

Corollary: don't "skip the dangerous op entirely" if doing so orphans state with no UI home —
the credentials-into-connectors PR removed the Credentials tab, so skipping the global purge
would have left a workspace connector's company key unreachable. Keep the op, gate the authority.

- **subprocess realization** (`user-files-host-ops.ts`, the canary/CI path): shares host FS → read/delete the resolved `localDir` hostPath DIRECTLY. host-read uses only `lstat`/`readFile`/`readdir` (never a writable handle); a `safeJoin` rejects absolute + `..` AND re-checks the post-join path is inside the root; a symlink at the target → absent (don't follow agent-planted links). cleanup = `fs.rm(hostPath,{recursive,force})`. nfs kind → throw (mis-wired preset).
- **k8s realization** (`user-files-ops.ts`): no host FS → a SHORT-LIVED one-shot pod (`buildOneshotPod`) mounts the WHOLE export inline (`{nfs:{server,path:exportPath}}` at `/export`), operates on `/export/$SUBPATH`. SUBPATH/RELPATH ride in ENV (never spliced into a shell word) + re-validated single-segment (`^[A-Za-z0-9_-]+$`); cmd is `rm -rf -- "$EXPORT/$SUBPATH"`. host-read mounts `readOnly:true` and emits one bounded line over `pods/log` (`DIR <b64>`/`FILE <b64>`/`BIG`/`ABSENT`; base64 keeps binary intact, 1 MiB cap); `parseReadOutput` parses it. Pod is fenced+locked exactly like a runner: label `ax.io/plane:execution` (→ existing `sandbox-restrict` NetworkPolicy gives it the Filestore-IP NFS egress — NO new NetworkPolicy/RBAC needed; `pods create/delete/get`+`pods/log get` already granted) + a DISTINCT `app.kubernetes.io/component: ax-next-userfiles` so the runner orphan-sweep (selects `=ax-next-runner`) never touches it. Reuses `watchPodExit` (terminal phase) + `killPod` (idempotent delete). uid/gid 1000 matches the runner so it can write/read the subtree the runner created.
- **Cross-tenant safety (§9, the load-bearing property):** deleting agent A touches ONLY A's subtree. Proven by tests in BOTH providers (subprocess: seed A+B, delete A, assert B's files survive; k8s: assert the created pod's `SUBPATH` env == exactly the deleted agentId). The single validated agentId segment is what guarantees it — same gate as the mount itself.
- **HOST-READ SYMLINK CONFINEMENT (security review caught this — HIGH cross-tenant disclosure).** A lexical relPath guard (reject absolute/`..`) is NECESSARY BUT NOT SUFFICIENT — and rejecting a symlink ONLY at the FINAL path component is the bug. The agent can plant an INTERMEDIATE symlink INSIDE its own subtree (`/workspace/escape -> /export/<other-agentId>` or `-> /`) and a naive read follows it into a sibling tenant's files. Fix = REALPATH-confine (resolve EVERY component) the target to the agent's OWN subtree BEFORE reading/listing:
  - subprocess (`user-files-host-ops.ts`): `realRoot=fs.realpath(root); realTarget=fs.realpath(target); if (realTarget!==realRoot && !realTarget.startsWith(realRoot+sep)) return absent;` THEN open the final file with `O_NOFOLLOW` (`fs.open(target, O_RDONLY|O_NOFOLLOW)`, read via the fd) to shrink the realpath→open TOCTOU window. Dir listing drops symlink children automatically (`readdir withFileTypes` uses lstat → a symlink reports neither isFile nor isDirectory).
  - k8s reader shell script: confine to the per-agent subPath, NOT just `$EXPORT` (a symlink could still cross into a sibling's subtree on the SAME export): `base="$EXPORT/$SUBPATH"; real=$(realpath -- "$target"); case "$real" in "$realbase"|"$realbase"/*) ;; *) echo ABSENT; exit;; esac`; skip symlink children in the listing (`[ -L "$entry" ] && continue`). Pass SUBPATH + a subtree-relative RELPATH SEPARATELY via env (never pre-join, never splice into a shell word).
  - Regression test for EACH path: an intermediate symlink → sibling subPath AND → absolute `/` must yield absent, not disclose. The subprocess tests use the REAL on-disk FS (it shares the host FS); the k8s test asserts the generated script carries the realpath-confine + symlink-skip (a unit test can't run a real NFS pod — the kind walk exercises it live).

## AgentContext.source across the IPC boundary (TASK-181, host-only, unforgeable)

The happy-path runner-COMPLETED `chat:end` is fired by the IPC server from a ctx
**reconstructed off the session-token resolution** (`ipc-server/listener.ts:144` +
the TCP twin `ipc-http/listener.ts`). So a per-ctx field consumed there must ride the
session record, not the inbound frame. The full host-only chain for `source`
(`'routine'|'user'`, mirrors the earlier `conversationId` chain exactly):

  routines `fire.ts` stamps `ctx.source='routine'`
   → orchestrator `runAgentInvoke` forwards `ctx.source` into `owner.source` on
     `sandbox:open-session` (conditional-spread: key ABSENT for user turns,
     exactOptionalPropertyTypes)
   → `@ax/sandbox-protocol` `OpenSessionInputSchema.owner.source` (`z.enum(['routine','user']).optional()`)
   → both backends (`sandbox-subprocess`/`sandbox-k8s` `open-session.ts`) forward
     `input.owner.source` into their local `SessionCreateInput.owner.source`
   → both session stores (`session-inmemory`/`session-postgres`) persist it on the
     record (postgres: additive `ALTER TABLE … ADD COLUMN IF NOT EXISTS source TEXT`,
     read back narrowed via `normalizeSource`; unknown stored value ⇒ null ⇒ user)
   → `session:resolve-token` returns `source` (added to `SessionResolveTokenOutput` +
     its return-schema in BOTH stores; both `validateOwner`s accept `'routine'|'user'|null`
     and reject anything else)
   → `@ax/ipc-core` `authenticate()` carries it onto `AuthResult.source`
   → both listeners stamp `...(auth.source !== null ? { source: auth.source } : {})`
   → `@ax/memory-strata`'s two `chat:end` subscribers early-return on `ctx.source==='routine'`.

**SECURITY (load-bearing):** `source` is HOST-ONLY — deliberately absent from
`@ax/ipc-protocol` and never read off the chat-end frame body. The runner controls only
its bearer token, which merely *selects which session record to read* — it cannot set the
stored `source`. The runner never calls `session:create`/`sandbox:open-session` (host-side
only). Tests assert a runner smuggling `source:'routine'` in the `/event.chat-end` body
does NOT flip `ctx.source` (`ipc-server/__tests__/source-propagation.test.ts` +
`ipc-http/__tests__/auth-ctx-propagation.test.ts`). The `EventChatEndSchema` is
`z.object({outcome})` which strips unknown keys anyway.

**Drift-guard gotcha:** adding a required field to `SessionResolveTokenOutput` (in either
store) forces an update to that store's `return-schemas.test.ts` fixture (the `: Type`
annotation catches it at compile; the `.toEqual(full)` round-trip catches a schema that
strips it). Also: every `resolveToken().toEqual({...})` in store/plugin tests (both
backends) is exact-match and needs the new `source:` key.

## 2026-06-08 (TASK-183) — `routines:upsert-default` now flips a default's GLOBAL `enabled`
- **The TASK-178 note above ("flipped post-walk via `routines:upsert-default`") was aspirational, not real, until TASK-183.** `store.ts upsertDefault` hard-coded `enabled` to `true` on INSERT and **omitted `enabled` from the `ON CONFLICT (name) DO UPDATE SET`** — so re-upserting the seeded `skill-reflection` default (seeded `enabled=false`) never changed its global kill-switch. The TASK-180 walk had to flip it with raw SQL. Fix: thread an optional `enabled?: boolean` through `UpsertDefaultInput` → store; INSERT uses `input.enabled ?? true`; the UPDATE path adds `enabled = EXCLUDED.enabled` **only when the caller passed `enabled`**.
- **Why conditional, not unconditional, on the UPDATE path:** a plain routine-spec re-upsert carries NO `enabled` (it's an operator-only out-of-band flag, NOT a routine-markdown frontmatter field — `parse-routine.ts` never produces it). If `enabled = EXCLUDED.enabled` were always in the SET, every spec edit would silently re-enable a deliberately-off default. So the flip is opt-in per call.
- **kysely conditional `ON CONFLICT … DO UPDATE SET` fragment pattern:** build `const frag = input.enabled === undefined ? sql\`\` : sql\`enabled = EXCLUDED.enabled,\`;` then interpolate `${frag}` as one more SET line (keep the trailing comma INSIDE the fragment, place it before a line that always exists like `source_md = …`). Empty `sql\`\`` interpolates to nothing — clean way to make a SET clause conditional without two separate query strings. The boolean value is bound (`${insertEnabled}`), the column name is a static literal — no injection surface.
- **The hook input type already declared `enabled?: boolean` (TASK-178 anticipated it) but it was unwired** — handler never read `input.enabled`, store never accepted it. So TASK-183 is "wire an existing-but-dead contract field", NOT a hook-surface change → no boundary review.
- **The admin HTTP route (`@ax/routines-admin-routes`) still rejects `enabled`.** Its `upsertDefaultBodySchema` is `.strict({ sourceMd })` with a comment saying "extra fields reject … until the upsert hook itself accepts one." The hook now accepts one, but exposing it through the admin route (which admins / what UI) is a separate operator-UI decision — left out of TASK-183 scope, flagged as a follow-up. Flipping the global kill-switch is correctly reachable via the hook (walk / operator path), which is the acceptance bar.

## 2026-06-22 (filestore /workspace) — NFS subPath is root-owned; the non-root runner needs a chown init
- **Root cause of "agent can't write /workspace" on real Filestore:** the design's §14 open risk (`2026-06-07-filestore-user-files-design.md`) resolved the bad way. The k8s runner mounts the per-agent durable tier as an inline `nfs:` volume with `subPath=<agentId>`; the kubelet AUTO-CREATES that subPath dir as `root:root` on first mount. The runner is non-root (`runAsUser:1000`), so it gets `Permission denied` writing `/workspace`. This was never caught because **no Filestore instance ever existed** — `gcloud filestore instances list` was empty; the feature shipped untested against real NFS.
- **`fsGroup:1000` does NOT fix it on NFS.** Verified empirically on Google Filestore + gVisor: a pod-level `securityContext.fsGroup:1000` leaves the subPath dir `root:root` (the NFS volume plugin doesn't apply fsGroup recursive-chown). So the existing services-only fsGroup (pod-spec.ts) is irrelevant to the user-files mount.
- **The fix (pod-spec.ts `buildUserFilesChownInit`):** a root init container, one per WRITABLE `role:'user-files'` mount, that mounts the SAME volume + SAME `subPath` (so it sees ONLY that agent's subtree — no cross-tenant exposure) and `chown 1000:1000`s the mount point before the runner starts. Runs `runAsUser:0` + `capabilities:{drop:['ALL'],add:['CHOWN']}` (chown(2) to a new owner needs CAP_CHOWN; everything else dropped, rootfs read-only). Owner+target via env, never spliced into the shell word (matches user-files-ops.ts discipline). Placed AFTER `sdk-scaffold` so it stays init[0] (service-sidecar tests assert that).
- **Why root works here:** Google Cloud Filestore exports `no_root_squash` by default, so a root client CAN chown. Verified with a throwaway pod (mount export, `chown 1000:1000 <dir>`, then write as uid 1000 → OK). If a future NFS backing uses root_squash, this init fails loud (pod won't start) rather than silently leaving it unwritable.
- **`ax-next-runners` namespace allows root containers** (no `restricted` PodSecurity enforcement) — a `runAsUser:0` verify pod scheduled fine. The chown init relies on that.
- **Deploy wiring:** `/workspace` only exists when `AX_FILESTORE_SERVER` is stamped, which needs `sandbox.filestore.server` in helm values (`presets/k8s/src/index.ts` loads `@ax/workspace-filestore` only then). Chart default is `server:""` = disabled. Also needs the running image to include #340 (TASK-175, 2026-06-08) — the pre-#340 agentId regex `^[a-z0-9-]+$` rejected real `agt_<base64url>` ids and left the mount inert.
