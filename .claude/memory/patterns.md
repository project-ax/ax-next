# Patterns

## Patterns (do these)

- `2026-04-23` — When adding a new service hook, fill in the boundary-review block in the PR description (alternate impl / leaky field names / subscriber risk / wire surface). If you can't name an alternate impl, it probably shouldn't be a hook yet.
- `2026-04-23` — When scaffolding a plugin that touches fs/net/process/untrusted input, drop a `SECURITY.md` in the package directory at scaffold time, not later. Week 3's `packages/storage-sqlite/SECURITY.md` is the template.
- `2026-04-23` — For ESM "am I the main module?" checks, use `import { pathToFileURL } from 'node:url'` and compare `import.meta.url === pathToFileURL(process.argv[1]).href`. `require.main === module` doesn't exist under ESM.
- `2026-04-29` — When reading a Claude Agent SDK jsonl by sessionId, glob `<HOME>/.claude/projects/*/<sessionId>.jsonl` rather than reconstructing the encoded-cwd directory. The SDK encodes `realpath(cwd)`, not the literal cwd, so any symlink (e.g. macOS `/var` → `/private/var`) breaks reconstruction. SessionId is unique across project dirs.
- `2026-04-29` — When parsing SDK jsonl, whitelist turn-bearing entry types (`user`/`assistant`/`tool`) rather than blacklisting bookkeeping types. The SDK adds new bookkeeping kinds (`queue-operation`, `last-prompt`, `attachment`) over time; a blacklist drifts.
- `2026-05-03` — When a PR wires a new plugin into the k8s preset, the chart-shape test (`deploy/charts/ax-next/__tests__/env-shape.test.ts`) is the contract that catches missed env-var stamping. The test scans `presets/k8s/src/index.ts` for `env.*` reads and asserts both directions against the rendered deployment. Run it; if it fails, the chart's env block is incomplete — fix in the same PR.
- `2026-05-03` — Subcommands that need an HTTP front door should register routes via `http:register-route` against `@ax/http-server`, not spin up their own `http.createServer`. Mirror the channel-web / auth-oidc / teams pattern: a plugin with `manifest.calls: ['http:register-route']`, routes registered in `init()`. Multiple listeners on the same pod cause `EADDRINUSE` collisions and bypass the http-server's CSRF / body-cap subscribers.
- `2026-05-24` — Adding a NEW field to `PluginManifestSchema` must be `.optional()` (NOT `.default([])`) if you don't want to touch every existing in-code manifest. `PluginManifest = z.infer<schema>` uses the schema's *output* type, and a `.default([])` field is **required** in the output type — so all ~20 hand-authored manifest literals fail tsc with "Property X is missing". `.optional()` makes it `X?:` in the inferred type (additive, zero churn); have `bootstrap` treat the absent value as empty (`manifest.optionalCalls?.map(...) ?? []`). This is why `calls`/`registers`/`subscribes` have `.default([])` yet don't error — every manifest already spells them out explicitly.
- `2026-05-24` — `pnpm` is NOT on the orchestrated-agent PATH; only `node` + `corepack` are (under `~/.nvm/versions/node/<v>/bin`). Run pnpm as `corepack pnpm <args>`. But the root `test` script does `pnpm -r run test` internally, and that nested invocation can't find `pnpm` → `sh: 1: pnpm: not found`. Fix: drop a 2-line wrapper (`#!/usr/bin/env bash\nexec corepack pnpm "$@"`) into a tmp dir and prepend it to PATH so nested `pnpm` resolves. (Alternatively `corepack enable --install-directory $(dirname $(which node))` installs a real `pnpm` shim on PATH.)
- `2026-05-24` — When a hook payload schema is duplicated across plugins (and drifting), the sanctioned fix is a pure `@ax/*-protocol` package: zod schemas + inferred types, NO `@ax/core` dep, mirror `packages/workspace-protocol` exactly (package.json/tsconfig/vitest.config), register in root `tsconfig.json` `references`, and add `'!@ax/<name>'` to the `no-restricted-imports` group in `eslint.config.mjs` (plus the doc-comment + trailing message string). Backends `safeParse` the shared schema at their boundary (keep their own `PluginError` wrapping); a constructor-only consumer (the orchestrator) imports the inferred TYPES only (type-only import). Converge drifted variants UP to the stricter one. `@ax/sandbox-protocol` (ARCH-5) joined `@ax/ipc-protocol` / `@ax/workspace-protocol` this way.
- `2026-05-24` — Same `@ax/*-protocol` wiring recipe also applies when relocating **git-vocabulary hook-bus payload TYPES** out of the storage-neutral `@ax/core` kernel (ARCH-3 → `@ax/workspace-bundle-protocol`, holding the 4 optional bundle-wire types `WorkspaceApplyBundle*`/`WorkspaceExportBaselineBundle*`). TWO differences from the zod `*-protocol` packages: (1) it exports **pure TS types, NO zod** — the contract is an in-process `bus.call<In,Out>`/`registerService<In,Out>` generic, not an untrusted wire (the untrusted runner↔host wire is already `@ax/ipc-protocol`, which DOES zod-validate; git backends re-derive `baselineCommit` from their own scratch repo), so zod here would be dead validation; (2) it **DOES depend on `@ax/core`** (type-only) because the bundle types reference neutral `WorkspaceVersion`/`WorkspaceApplyOutput`. Decision rule for git-vocab on the workspace surface: *wire* shape → `@ax/ipc-protocol`; *hook-bus* type → its own git-runner `*-protocol` package; never `@ax/core` (I1 / arch §4.5 forbids sha/branch/bundle/ref/commit there). Optional-hook mechanism untouched — non-git backends register neither hook and never import the package.
- `2026-05-24` — Codex CLI **IS installed + authed in this orchestrated env** (`codex` on the nvm PATH, `~/.codex/auth.json` present, no `OPENAI_API_KEY` needed) — the ARCH-4/5 "codex unavailable" notes were per-run, re-check don't assume. Drive yolo-ship Phase 5 with `node "$(ls -d ~/.claude/plugins/cache/openai-codex/codex/*/ | sort -V | tail -1)scripts/codex-companion.mjs" review --wait --scope branch --base main` from the worktree. The companion's OWN verification shell lacks `corepack`/`pnpm`/`tsc` on PATH; it self-recovers (prepends `~/.nvm/.../bin`, calls `./node_modules/.bin/tsc`/`eslint`) and completes fine.
- `2026-05-24` — **A concurrent yolo-ship agent in the SHARED main checkout overwrote `.claude/memory/decisions.md`**, silently dropping appended rows (it rewrote the file with its own newest entries). When multiple auto-ship agents run in parallel, writing memory to the shared main checkout races. Safer in orchestrated/parallel mode: write the session's memory rows into the **worktree's branch copy** and commit them with the PR — branch-isolated, survives the contention, lands on merge. (Reverses the decisions-45/52 "write to main checkout" guidance specifically for the parallel-agents case.)
- `2026-05-24` — The env lacks `git lfs`, so `@ax/agent-claude-sdk-runner`'s 24 `git-workspace` tests fail locally with `git: 'lfs' is not a git command` — a local-env gap, NOT a regression (CI has git-lfs). Build the whole workspace before running any package's tests — sibling devDeps resolve via their `dist/` `exports`, so an unbuilt dep fails with `Failed to resolve entry for package "@ax/..."`.
- `2026-05-24` — **Testcontainers tests are UNRUNNABLE locally in the orchestrated env** — `/var/run/docker.sock` is `root:docker` and the agent user (`vpulim`) is NOT in the `docker` group (no passwordless sudo, and `dangerouslyDisableSandbox` doesn't help — it's a host socket permission, not the sandbox). Any test that boots a `PostgreSqlContainer` (preset-k8s `acceptance.test.ts` Phase D/F + ARCH-8 `prod-bootstrap.test.ts`) fails with `Could not find a working container runtime strategy` — a local-env gap, NOT a regression (CI has Docker; these suites are green on main). To verify the parts you DO control without Docker: extract the non-DB logic (config build, plugin filter/splice, manifest shape, duplicate-registrant check, env-loader) into a throwaway `.mjs` run with `node` against the built `dist/` — proves the assembly is well-formed; the actual `bootstrap()`-against-Postgres step is CI's job.
- `2026-05-24` — **SHARED-WORKING-COPY HAZARD: parallel auto-ship agents can all operate on the SAME main checkout `/home/vpulim/dev/ai/ax-next` and stomp each other's `git checkout`.** ARCH-8 created its branch with `git checkout -b` in the main copy; a concurrent ARCH-2 agent then `git checkout`ed ITS branch in the same copy, so HEAD silently moved off ARCH-8's branch and `git diff main...HEAD` showed ARCH-2's files. The fix: as soon as you notice you're sharing the main checkout (or proactively at Phase 0 if `EnterWorktree` errored with "already isolated"), create a DEDICATED `git worktree add .claude/worktrees/<TASK> <branch>` and do ALL remaining work there with absolute paths + `git -C <worktree>`. Your commit is safe on your branch ref regardless of HEAD; verify with `git diff main...<your-branch>` (three-dot, merge-base) NOT `main...HEAD`. Never `git stash`/`commit -a` in the shared copy — you'll capture another agent's files.
- `2026-05-24` — When driving CI green, the GitHub API for a freshly-pushed commit can serve **wildly inconsistent state across replicas**: `gh run view <id> --json status`, the PR `statusCheckRollup`, and even the per-job `/actions/jobs/<id>` endpoint each flapped between `in_progress` and `completed/success` *for the same commit at the same wall-clock time* — a forward replica says done while a lagging one (and `gh pr view --json mergeStateStatus`) still says UNSTABLE. A single "success" read (or even a Monitor that confirms 3-4× on ONE endpoint) is NOT trustworthy; the monitor process and the foreground shell hit different replicas and disagree. Trust the **immutable** signal: a job's `completed_at` timestamp (null until truly done, never reverts) and the test job's own *step* progression (`Test (affected packages + dependents)` → `Test (full suite)` → `pnpm audit`). Cross-check two endpoints and require agreement; treat any disagreement as "not done." Don't post a premature `CI green` heartbeat off a flapping read.
- `2026-05-24` — The CI `test` job runs a 3-step sequence: `Test (affected packages + dependents)` → `Test (full suite)` (often `skipped` when affected-scope already covered it) → `pnpm audit --audit-level moderate`. The whole job took ~6–14 min on `@ax/core`-touching diffs. The CI runner HAS git-lfs, so `agent-claude-sdk-runner` `git-workspace.test.ts` (which `git lfs install`s) passes there even though it fails locally on hosts without git-lfs — that local failure is environmental, not a regression.

## Anti-Patterns (don't do these)

- `2026-04-23` — Don't put backend-specific field names in hook payloads (`sha`, `bucket`, `pod_name`, `table`, `rowid`). If the name only makes sense for one impl, rename before subscribers depend on it.
- `2026-04-23` — Don't land "we'll wire it later" infrastructure. Either a live caller reaches it in the same PR or it doesn't merge. Symmetric API pairs (`get`/`set`) with a documented near-term consumer are the narrow exception.
- `2026-05-24` — Promoting a hook from absent to a REQUIRED `manifest.calls` entry can break canary acceptance harnesses that deliberately DROP the producer plugin. ARCH-2 expanded `@ax/ipc-server`/`@ax/ipc-http` required `calls` (spread from `@ax/ipc-core`'s `DISPATCHER_DEPENDENCIES`) to include `workspace:read` + `conversations:store-runner-session`. `presets/k8s/src/__tests__/acceptance.test.ts` + `multi-tenant-acceptance.test.ts` build "preset-equivalent" sets via `createK8sPlugins(cfg)` then `.filter(p => !PLUGINS_TO_DROP.has(name))` — dropping `@ax/workspace-git` (local backend) and `@ax/conversations` to stay postgres-free — so `bootstrap`'s `verifyCalls` then fails `'@ax/ipc-server' declares calls:['workspace:read'] but no plugin registers it`. Fix: a tiny no-op `createDispatcherDepsStubPlugin(hooks[])` registered alongside `createIpcServerPlugin()`, passing ONLY the missing subset (git-protocol backend keeps `@ax/workspace-git-server` which still registers `workspace:read`, so those lists only need the `conversations:store-runner-session` stub; local-backend lists need both). Passing a hook a kept plugin already registers trips `checkDuplicateRegisters`' `duplicate-service`. The real preset (`prod-bootstrap`/`preset.test.ts`) is fine — it keeps the real producers. When you widen a transport/plugin's required calls, grep the preset canaries for `PLUGINS_TO_DROP` + the new producer and stub the gap in the SAME PR.
- `2026-05-24` — This env has NO Docker/testcontainers: `presets/k8s` tests that `new PostgreSqlContainer(...).start()` fail locally with `Could not find a working container runtime strategy` (acceptance Phase D / attachments / artifact_publish / Task 21, all of `prod-bootstrap.test.ts`). Like the git-lfs gap, this is environmental, not a regression — CI has Docker. To verify a preset change locally, run the non-testcontainer canaries by name (`-t "boots a preset-equivalent"`, `-t "Phase 3 canary: workspace.commit-notify"`, the whole `multi-tenant-acceptance.test.ts`, and `preset.test.ts` static wiring) and confirm the failures that remain are ONLY `container runtime` errors.
