# Patterns

## Patterns (do these)

- `2026-05-25` — **For a guard test that must run on EVERY PR regardless of which package changed (e.g. a Dockerfile-content/repo-root-file assertion that isn't owned by any pnpm package), put it in `scripts/__tests__/*.test.js`.** CI's PR test step gates package tests behind `pnpm --filter "...[$BASE_SHA]"` (affected + dependents), but it then runs `pnpm test:eslint-rules` and `pnpm test:scripts` (`vitest run --root scripts`) UNCONDITIONALLY (ci.yml lines ~60-61). So a guard in `@ax/agent-claude-sdk-runner` would be SKIPPED on a PR that touches only `container/agent/Dockerfile` (the ARCH-11 path-reach/affected-selection trap), but a guard in `scripts/__tests__/` always runs. Resolve the target file repo-root-relative from `import.meta.url` (`join(dirname(fileURLToPath(import.meta.url)), '..', '..', ...)`), NOT cwd, so it's worktree-safe. The `scripts/` vitest root has NO config — `vitest run --root scripts` picks up `**/*.test.js` by default; existing example `scripts/__tests__/memory-write-target.test.js`. (TASK-19's `dockerfile-home-bin-path.test.js`.)
- `2026-05-25` — **The agent's RUNTIME `$HOME` is `/permanent` (the git-bundled workspace tier), NOT the image's `/home/axagent`.** `agent-claude-sdk-runner/src/main.ts` sets `HOME: env.workspaceRoot` (default `/permanent`, env.ts:105) + `cwd: env.workspaceRoot` on the SDK subprocess. And the SDK's Bash tool runs a **non-interactive** shell whose PATH comes ENTIRELY from the explicit `query({options:{env}})` allowlist — `proxy-startup.ts` forwards `process.env.PATH` and comments that without it the Bash tool's `ls` exits 127 (proof: no rc file is sourced, else a default PATH would exist). Consequences: (1) a `.bashrc` baked into the image is NOT read by the agent's shell — to actually affect the agent's PATH you must prepend to the forwarded env (the `buildToolCacheEnv`/`buildPythonVenvEnv`/`buildHomeBinEnv` layers in main.ts's env literal); (2) "persists between sessions" = under `/permanent` (bundled + restored), so `$HOME/bin` = `/permanent/bin`. The env literal can't reference its own earlier keys, so to layer a new PATH segment relative to the venv bin, compute the venv env in a const BEFORE the `query()` call and feed its `.PATH` (or the base PATH when venv off) into the new layer spread LAST. (TASK-19.) **SECURITY (codex Phase-5): APPEND a model-writable persisted dir to PATH, never PREPEND** — `$HOME=/permanent` is restored across sessions, so a leading entry lets an injected `/permanent/bin/git` (etc.) persistently shadow trusted image/venv tools (I5 violation). The venv bin DOES prepend (correctly — it's seeded + session-ephemeral, and python/pip must win), but `$HOME/bin` appends so trusted bins win on name collisions.
- `2026-04-23` — When adding a new service hook, fill in the boundary-review block in the PR description (alternate impl / leaky field names / subscriber risk / wire surface). If you can't name an alternate impl, it probably shouldn't be a hook yet.
- `2026-04-23` — When scaffolding a plugin that touches fs/net/process/untrusted input, drop a `SECURITY.md` in the package directory at scaffold time, not later. Week 3's `packages/storage-sqlite/SECURITY.md` is the template.
- `2026-04-23` — For ESM "am I the main module?" checks, use `import { pathToFileURL } from 'node:url'` and compare `import.meta.url === pathToFileURL(process.argv[1]).href`. `require.main === module` doesn't exist under ESM.
- `2026-04-29` — When reading a Claude Agent SDK jsonl by sessionId, glob `<HOME>/.claude/projects/*/<sessionId>.jsonl` rather than reconstructing the encoded-cwd directory. The SDK encodes `realpath(cwd)`, not the literal cwd, so any symlink (e.g. macOS `/var` → `/private/var`) breaks reconstruction. SessionId is unique across project dirs.
- `2026-04-29` — When parsing SDK jsonl, whitelist turn-bearing entry types (`user`/`assistant`/`tool`) rather than blacklisting bookkeeping types. The SDK adds new bookkeeping kinds (`queue-operation`, `last-prompt`, `attachment`) over time; a blacklist drifts.
- `2026-05-03` — When a PR wires a new plugin into the k8s preset, the chart-shape test (`deploy/charts/ax-next/__tests__/env-shape.test.ts`) is the contract that catches missed env-var stamping. The test scans `presets/k8s/src/index.ts` for `env.*` reads and asserts both directions against the rendered deployment. Run it; if it fails, the chart's env block is incomplete — fix in the same PR.
- `2026-05-03` — Subcommands that need an HTTP front door should register routes via `http:register-route` against `@ax/http-server`, not spin up their own `http.createServer`. Mirror the channel-web / auth-oidc / teams pattern: a plugin with `manifest.calls: ['http:register-route']`, routes registered in `init()`. Multiple listeners on the same pod cause `EADDRINUSE` collisions and bypass the http-server's CSRF / body-cap subscribers.
- `2026-05-24` — Adding a NEW field to `PluginManifestSchema` must be `.optional()` (NOT `.default([])`) if you don't want to touch every existing in-code manifest. `PluginManifest = z.infer<schema>` uses the schema's *output* type, and a `.default([])` field is **required** in the output type — so all ~20 hand-authored manifest literals fail tsc with "Property X is missing". `.optional()` makes it `X?:` in the inferred type (additive, zero churn); have `bootstrap` treat the absent value as empty (`manifest.optionalCalls?.map(...) ?? []`). This is why `calls`/`registers`/`subscribes` have `.default([])` yet don't error — every manifest already spells them out explicitly.
- `2026-05-24` — `pnpm` is NOT on the orchestrated-agent PATH; only `node` + `corepack` are (under `~/.nvm/versions/node/<v>/bin`). Run pnpm as `corepack pnpm <args>`. But the root `test` script does `pnpm -r run test` internally, and that nested invocation can't find `pnpm` → `sh: 1: pnpm: not found`. Fix: drop a 2-line wrapper (`#!/usr/bin/env bash\nexec corepack pnpm "$@"`) into a tmp dir and prepend it to PATH so nested `pnpm` resolves. (Alternatively `corepack enable --install-directory $(dirname $(which node))` installs a real `pnpm` shim on PATH.)
- `2026-05-24` — When a hook payload schema is duplicated across plugins (and drifting), the sanctioned fix is a pure `@ax/*-protocol` package: zod schemas + inferred types, NO `@ax/core` dep, mirror `packages/workspace-protocol` exactly (package.json/tsconfig/vitest.config), register in root `tsconfig.json` `references`, and add `'!@ax/<name>'` to the `no-restricted-imports` group in `eslint.config.mjs` (plus the doc-comment + trailing message string). Backends `safeParse` the shared schema at their boundary (keep their own `PluginError` wrapping); a constructor-only consumer (the orchestrator) imports the inferred TYPES only (type-only import). Converge drifted variants UP to the stricter one. `@ax/sandbox-protocol` (ARCH-5) joined `@ax/ipc-protocol` / `@ax/workspace-protocol` this way.
- `2026-05-24` — Same `@ax/*-protocol` wiring recipe also applies when relocating **git-vocabulary hook-bus payload TYPES** out of the storage-neutral `@ax/core` kernel (ARCH-3 → `@ax/workspace-bundle-protocol`, holding the 4 optional bundle-wire types `WorkspaceApplyBundle*`/`WorkspaceExportBaselineBundle*`). TWO differences from the zod `*-protocol` packages: (1) it exports **pure TS types, NO zod** — the contract is an in-process `bus.call<In,Out>`/`registerService<In,Out>` generic, not an untrusted wire (the untrusted runner↔host wire is already `@ax/ipc-protocol`, which DOES zod-validate; git backends re-derive `baselineCommit` from their own scratch repo), so zod here would be dead validation; (2) it **DOES depend on `@ax/core`** (type-only) because the bundle types reference neutral `WorkspaceVersion`/`WorkspaceApplyOutput`. Decision rule for git-vocab on the workspace surface: *wire* shape → `@ax/ipc-protocol`; *hook-bus* type → its own git-runner `*-protocol` package; never `@ax/core` (I1 / arch §4.5 forbids sha/branch/bundle/ref/commit there). Optional-hook mechanism untouched — non-git backends register neither hook and never import the package.
- `2026-05-24` — Codex CLI **IS installed + authed in this orchestrated env** (`codex` on the nvm PATH, `~/.codex/auth.json` present, no `OPENAI_API_KEY` needed) — the ARCH-4/5 "codex unavailable" notes were per-run, re-check don't assume. Drive yolo-ship Phase 5 with `node "$(ls -d ~/.claude/plugins/cache/openai-codex/codex/*/ | sort -V | tail -1)scripts/codex-companion.mjs" review --wait --scope branch --base main` from the worktree. The companion's OWN verification shell lacks `corepack`/`pnpm`/`tsc` on PATH; it self-recovers (prepends `~/.nvm/.../bin`, calls `./node_modules/.bin/tsc`/`eslint`) and completes fine.
- `2026-05-24` — **CANONICAL (TASK-7): agents write + commit `.claude/memory/` ONLY in their own worktree/branch copy, never the shared main checkout.** A concurrent yolo-ship agent in the SHARED main checkout once overwrote `.claude/memory/decisions.md`, silently dropping appended rows (it rewrote the file with its own newest entries); parallel auto-ship agents writing the shared checkout race. The branch copy is race-free: memory is tracked so `git worktree add` carries it in, rows land on PR merge via the serialized queue, and a true clash becomes a visible git merge conflict, not a silent drop. This is now the standardized rule in CLAUDE.md "Codex Memory Bootstrap" + the `claude-memory` skill, and it **supersedes** the older decisions-45/52/55/60/67/73 "write decisions.md to the main checkout" guidance (which predated memory being committed, when a fresh worktree's copy was empty — it no longer is). Helper: `scripts/memory-write-target.sh` prints the right dir for your tree and (`--check`) fails if you're in the shared main checkout while linked worktrees exist.
- `2026-05-24` — The env lacks `git lfs`, so `@ax/agent-claude-sdk-runner`'s 24 `git-workspace` tests fail locally with `git: 'lfs' is not a git command` — a local-env gap, NOT a regression (CI has git-lfs). Build the whole workspace before running any package's tests — sibling devDeps resolve via their `dist/` `exports`, so an unbuilt dep fails with `Failed to resolve entry for package "@ax/..."`.
- `2026-05-24` — **Testcontainers tests are UNRUNNABLE locally in the orchestrated env** — `/var/run/docker.sock` is `root:docker` and the agent user (`vpulim`) is NOT in the `docker` group (no passwordless sudo, and `dangerouslyDisableSandbox` doesn't help — it's a host socket permission, not the sandbox). Any test that boots a `PostgreSqlContainer` (preset-k8s `acceptance.test.ts` Phase D/F + ARCH-8 `prod-bootstrap.test.ts`) fails with `Could not find a working container runtime strategy` — a local-env gap, NOT a regression (CI has Docker; these suites are green on main). To verify the parts you DO control without Docker: extract the non-DB logic (config build, plugin filter/splice, manifest shape, duplicate-registrant check, env-loader) into a throwaway `.mjs` run with `node` against the built `dist/` — proves the assembly is well-formed; the actual `bootstrap()`-against-Postgres step is CI's job.
- `2026-05-24` — **SHARED-WORKING-COPY HAZARD: parallel auto-ship agents can all operate on the SAME main checkout `/home/vpulim/dev/ai/ax-next` and stomp each other's `git checkout`.** ARCH-8 created its branch with `git checkout -b` in the main copy; a concurrent ARCH-2 agent then `git checkout`ed ITS branch in the same copy, so HEAD silently moved off ARCH-8's branch and `git diff main...HEAD` showed ARCH-2's files. The fix: as soon as you notice you're sharing the main checkout (or proactively at Phase 0 if `EnterWorktree` errored with "already isolated"), create a DEDICATED `git worktree add .claude/worktrees/<TASK> <branch>` and do ALL remaining work there with absolute paths + `git -C <worktree>`. Your commit is safe on your branch ref regardless of HEAD; verify with `git diff main...<your-branch>` (three-dot, merge-base) NOT `main...HEAD`. Never `git stash`/`commit -a` in the shared copy — you'll capture another agent's files.
- `2026-05-24` — When driving CI green, the GitHub API for a freshly-pushed commit can serve **wildly inconsistent state across replicas**: `gh run view <id> --json status`, the PR `statusCheckRollup`, and even the per-job `/actions/jobs/<id>` endpoint each flapped between `in_progress` and `completed/success` *for the same commit at the same wall-clock time* — a forward replica says done while a lagging one (and `gh pr view --json mergeStateStatus`) still says UNSTABLE. A single "success" read (or even a Monitor that confirms 3-4× on ONE endpoint) is NOT trustworthy; the monitor process and the foreground shell hit different replicas and disagree. Trust the **immutable** signal: a job's `completed_at` timestamp (null until truly done, never reverts) and the test job's own *step* progression (`Test (affected packages + dependents)` → `Test (full suite)` → `pnpm audit`). Cross-check two endpoints and require agreement; treat any disagreement as "not done." Don't post a premature `CI green` heartbeat off a flapping read.
- `2026-05-24` — The CI `test` job runs a 3-step sequence: `Test (affected packages + dependents)` → `Test (full suite)` (often `skipped` when affected-scope already covered it) → `pnpm audit --audit-level moderate`. The whole job took ~6–14 min on `@ax/core`-touching diffs. The CI runner HAS git-lfs, so `agent-claude-sdk-runner` `git-workspace.test.ts` (which `git lfs install`s) passes there even though it fails locally on hosts without git-lfs — that local failure is environmental, not a regression.

- `2026-05-24` — **TASK-12: the sandbox `git` binary does NOT trust the credential-proxy MITM CA via `NODE_EXTRA_CA_CERTS` / `SSL_CERT_FILE`.** Those two only steer Node's TLS (the SDK's undici fetch). The model's `git clone` runs the in-image `git` (libcurl/OpenSSL) via the Bash tool, which verifies the proxy cert against **`GIT_SSL_CAINFO`** (or `http.sslCAInfo` config) — pointed at the SAME CA path. k8s: stamp `GIT_SSL_CAINFO=/var/run/ax/proxy-ca/ca.crt` in `sandbox-k8s/pod-spec.ts` `proxyEnv` (alongside the two Node vars). subprocess: stamp `GIT_SSL_CAINFO=<caPath>` in `sandbox-subprocess/open-session.ts` (the per-session `ax-mitm-ca.pem`). It rides into the SDK subprocess automatically via proxy-startup's `GIT_` prefix allowlist (`ENV_ALLOWLIST_PREFIXES`) — no extra forwarding code needed, but add a regression test pinning the forward. Symptom when missing: `git clone` over the proxy dies `SSL certificate problem: unable to get local issuer certificate` (the CLI-1 walk-fail). The `echo-stub.mjs` test fixture must echo any new env key you assert on (it emits an explicit allowlist, absent keys read `undefined` not `null`).
- `2026-05-24` — **TASK-12 unsolved (followup): credential non-injection on git's Basic header.** With the CA fixed, the CLI-1 walk still 403'd from gitlab and the proxy audit showed `credentialInjected:false` on `GET /info/refs`. The framer's Basic-password substitution is correct in isolation (`request-framer.ts` `transformBasicAuthHead` round-trips `oauth2:ax-cred:<hex>` → real value, fully tested), so the bug is UPSTREAM of the pure function — most likely (a) git never sent the placeholder (no credential helper / URL userinfo wired into the Bash-tool git env), (b) the placeholder git sent wasn't registered for that session in `SharedCredentialRegistry` (so `replaceAll` is a no-op), or (c) a chunk-boundary case. Distinguishing needs a LIVE decrypted-bytes trace against the kind cluster (a worktree can't reproduce post-TLS-terminate bytes). A `.skip`'d pin lives in `credential-proxy/src/__tests__/request-framer.test.ts` ("TASK-12 part2"). To chase: `k8s-acceptance-loop` the git-clone walk, capture the proxy's `event.http-egress` audit, and check whether the request's Basic blob even contains an `ax-cred:` token before blaming the framer.

- `2026-05-24` — **ARCH-13 (returns-schema long tail): for a live-handle hook, the `.passthrough()` schema must NOT model the function field with `z.function()` — that WRAPS it in a validating proxy and breaks `returned === original` identity.** Modeling `http:register-route`'s `{ unregister(): void }` as `z.object({ unregister: z.function() }).passthrough()` made `.parse(out).unregister` a `[Function anonymous]` clone, not the same object — the drift-guard `expect(parsed.unregister).toBe(unregister)` failed. The correct posture (exactly what ARCH-6's `OpenSessionResultSchema` does for the live `handle`) is `z.object({ <serializable fields only> }).passthrough()` — model ONLY the storage-agnostic serializable keys and let the capability object ride through UNTOUCHED by reference. For `http:register-route` (output is ONLY the handle, no serializable field) that's a bare `z.object({}).passthrough()`. A capability object is not a data contract; don't try to validate it. Same applies to `eventbus:subscribe`'s `{ unsubscribe: () => void }`.

- `2026-05-25` — **ARCH-11: to keep TWO independent validators in sync without coupling them (a host zod schema + a sandbox-side hand-rolled re-validator that MUST NOT import the host package), use a SHARED GOLDEN-VECTORS JSON FIXTURE + a drift test on EACH side, not a cross-import.** The fixture is data (`{vectors:[{desc,core,schema:'accept'|'reject',runner:'accept'|'reject',value}]}`), so neither package imports the other. The schema-side test (in the package that owns the fixture) maps `Schema.safeParse` over the vectors; the runner-side test imports its own pure validator and reaches the fixture **read-only by repo-root-relative path** (`path.join(__dirname,'..','..','..','..','packages',...)` — the worktree-safe idiom, NOT cwd). Each vector declares an expected verdict PER SIDE so a legitimate asymmetry (the last-gate re-validator being STRICTER — defense in depth) is modeled without hiding drift; tag the security-critical vectors `core:true` and assert `schema===runner` for those. Two discipline points: (1) compute the expected verdicts by RUNNING both validators over the fixture in a throwaway `.mjs` BEFORE writing assertions (don't hand-wave them), and mutation-test the guard (break each validator, confirm the right side fails) to prove it actually detects drift; (2) add an independence self-check that greps the runner's `src` for an actual `import/require/from '@ax/<host-pkg>'` (anchored regex, NOT a bare mention — a cross-reference COMMENT naming the package is fine; exclude the guard file itself).
- `2026-05-25` — **ARCH-11: a path-reach test across packages has NO pnpm dependency edge, so CI's affected-package PR selection (`pnpm --filter "...[$BASE_SHA]"`) WON'T run it when only the OTHER package changes.** The runner-side drift test reads the sandbox-protocol fixture by path (deliberately no dep edge), so a PR touching only `packages/sandbox-protocol` selected sandbox-protocol but NOT the runner → the runner half of the guard would only run post-merge on main. Codex caught it. Fix WITHOUT a dep edge (which would defeat the independence): in the PR test step, `git diff --name-only "$BASE_SHA"...HEAD | grep -Eq '^packages/<other>/'` and, when matched, append `--filter @ax/<this-pkg>` to BOTH the affected-package list-probe and the run. Build the extra filter as a bash ARRAY (`extra=(--filter @ax/...)`, expand `"${extra[@]}"`) so shellcheck/actionlint stay clean (a bare `$extra` trips SC2086). Validate the workflow with the downloadable `actionlint` (the nested-worktree `pnpm lint` can't gate a YAML-only change). General rule: whenever a test relies on a *file-path reach* rather than a package import, ask "would `...[BASE_SHA]` select this test's package on a PR that changes only the reached file?" — if not, add a path-scoped CI inclusion.

## Anti-Patterns (don't do these)

- `2026-04-23` — Don't put backend-specific field names in hook payloads (`sha`, `bucket`, `pod_name`, `table`, `rowid`). If the name only makes sense for one impl, rename before subscribers depend on it.
- `2026-04-23` — Don't land "we'll wire it later" infrastructure. Either a live caller reaches it in the same PR or it doesn't merge. Symmetric API pairs (`get`/`set`) with a documented near-term consumer are the narrow exception.
- `2026-05-24` — Promoting a hook from absent to a REQUIRED `manifest.calls` entry can break canary acceptance harnesses that deliberately DROP the producer plugin. ARCH-2 expanded `@ax/ipc-server`/`@ax/ipc-http` required `calls` (spread from `@ax/ipc-core`'s `DISPATCHER_DEPENDENCIES`) to include `workspace:read` + `conversations:store-runner-session`. `presets/k8s/src/__tests__/acceptance.test.ts` + `multi-tenant-acceptance.test.ts` build "preset-equivalent" sets via `createK8sPlugins(cfg)` then `.filter(p => !PLUGINS_TO_DROP.has(name))` — dropping `@ax/workspace-git` (local backend) and `@ax/conversations` to stay postgres-free — so `bootstrap`'s `verifyCalls` then fails `'@ax/ipc-server' declares calls:['workspace:read'] but no plugin registers it`. Fix: a tiny no-op `createDispatcherDepsStubPlugin(hooks[])` registered alongside `createIpcServerPlugin()`, passing ONLY the missing subset (git-protocol backend keeps `@ax/workspace-git-server` which still registers `workspace:read`, so those lists only need the `conversations:store-runner-session` stub; local-backend lists need both). Passing a hook a kept plugin already registers trips `checkDuplicateRegisters`' `duplicate-service`. The real preset (`prod-bootstrap`/`preset.test.ts`) is fine — it keeps the real producers. When you widen a transport/plugin's required calls, grep the preset canaries for `PLUGINS_TO_DROP` + the new producer and stub the gap in the SAME PR.
- `2026-05-24` — This env has NO Docker/testcontainers: `presets/k8s` tests that `new PostgreSqlContainer(...).start()` fail locally with `Could not find a working container runtime strategy` (acceptance Phase D / attachments / artifact_publish / Task 21, all of `prod-bootstrap.test.ts`). Like the git-lfs gap, this is environmental, not a regression — CI has Docker. To verify a preset change locally, run the non-testcontainer canaries by name (`-t "boots a preset-equivalent"`, `-t "Phase 3 canary: workspace.commit-notify"`, the whole `multi-tenant-acceptance.test.ts`, and `preset.test.ts` static wiring) and confirm the failures that remain are ONLY `container runtime` errors.
- `2026-05-24` — **Digest-pinning an image dep (CLI-3): use the multi-arch INDEX digest, not a per-platform manifest digest.** `docker buildx imagetools inspect <ref>` prints `Digest: sha256:…` = the OCI index digest the tag points to → pin `image:tag@sha256:<index>` (keeps multi-arch + keeps the human-readable tag). `docker manifest inspect` (without `--verbose`) lists the *inner per-platform* manifest digests — pinning one of those would lock the image to a single arch. `docker manifest inspect --verbose` does expose the top-level `Descriptor.digest` (= the index), but `buildx imagetools inspect` is the clean one-liner. Daemon NOT required for either (they query the registry over HTTPS), so they work even when `docker run`/`build` is blocked (vpulim isn't in the `docker` group in this sandbox; no passwordless sudo to the socket).
- `2026-05-24` — **Resolving exact installable Debian apt versions without a container (CLI-3):** pull the signed binary Packages index directly — `curl -s https://deb.debian.org/debian/dists/bookworm/main/binary-amd64/Packages.gz | gunzip | awk '/^Package: (pkg)$/{p=$2}/^Version: /{if(p){print p": "$2;p=""}}'`. Gives the version actually in the `bookworm/main` pocket (what `apt-get install pkg=ver` would resolve). Caveat: a hard `=version` apt pin ROTS — when a point release supersedes it, the old `.deb` leaves the pocket and the build breaks. **TASK-9 closed this gap:** the `docker-build` lane in `ci.yml` now builds both `container/agent` and `container/git-server` images (buildx, push:false/load:false, path-gated to `container/**` + the workspace graph) on every change that touches them, so a rotted apt pin turns CI red instead of failing silently. Bump the pins in the Dockerfile when that happens. The credentialed-CLI security note (`docs/plans/2026-05-22-...-security-note.md`) is a point-in-time record of THAT PR's state; don't rewrite its "digest-pin is a tracked TODO" line when the follow-up lands — it's history, not a live spec.
- `2026-05-24` — ARCH-6 `returns`-schema rollout recipe (followed the credentials precedent): a hook's `returns` zod schema is a **per-registration in-process shape assertion** (NOT an inter-plugin wire contract — that's `@ax/ipc-protocol`), so it co-locates with the registering plugin's I/O interfaces. For shapes already in storage-neutral `@ax/core` (workspace read/list), the schema lives in core and all 3 backends import it (git-core, git-server client, test-harness mock). For two-backend hooks (session inmemory/postgres), each backend carries its OWN structurally-identical copy — same boundary cost as the duplicated I/O interfaces (I2). EVERY schema pairs with a **drift-guard test** (`return-schemas.test.ts`): round-trip a fully-populated INTERFACE-TYPED value through `.parse` and assert `toEqual` — a new required field fails at compile, a new optional one fails at runtime (the schema strips it → toEqual diverges). The credentials precedent (`CredentialsResolveOutputSchema` + its drift test) is the template.
- `2026-05-24` — ARCH-6: hooks that return a **live capability handle** (functions/Promises, e.g. `sandbox:open-session`'s `{runnerEndpoint, handle:{kill,exited}}`) MUST use a `.passthrough()` schema asserting only the serializable storage-agnostic field(s) — a strict `z.object` STRIPS the handle (hook-bus.ts:141-147 strips undeclared keys) and silently breaks the lifecycle. The `OpenSessionResultSchema` lives in `@ax/sandbox-protocol` (both backends already dep it); test it by asserting the handle survives by reference (`expect(parsed.handle).toBe(handle)`). Same trap for the workspace `apply`/`diff` delta (`changes[].contentBefore?/contentAfter?` are lazy fns) — those were DEFERRED, not schema'd, because read/list (IPC-reachable, clean data) were the priority subset and apply/diff need coordinated contract-test verification across backends.
- `2026-05-24` — ARCH-6: a zod schema used as a runtime VALUE (e.g. attaching `OpenSessionResultSchema` to `registerService({returns})`) makes `zod` a real dependency — plugins that previously had no zod dep (session-inmemory, session-postgres, sandbox-subprocess, sandbox-k8s) need `"zod": "^3.23.8"` added to package.json `dependencies` (NOT just devDeps); a bare `import type {ZodType} from 'zod'` ALSO needs zod resolvable, so even the type-only cast import forces the dep. Standard range across the repo is `^3.23.8` (core/agents/conversations/credentials all pin it). Casting the schema to `ZodType<O>` (`as unknown as ZodType<O>`) bridges zod's inferred `|undefined`/`|null`/passthrough shapes to the interface where `exactOptionalPropertyTypes` blocks direct assignability — the drift-guard test is the real enforcement.
- `2026-05-24` (TASK-11) — **The runner's "anything to ship this turn?" decision keys off the COMMIT RANGE `baseline..main`, NOT the working-tree staged diff.** `commitTurnAndBundle` did `git add -A` → `git diff --cached --quiet` → return null on a clean staged tree, conflating "no uncommitted changes" with "nothing to ship".
